🧠 แนวคิด & หลักการ

หลักการ 1: "อ่านโค้ดคนอื่นให้ขาด" คือทักษะที่ตลาดจ่ายแพงที่สุด

ปัญหาที่มันแก้: มือใหม่เจอ codebase เก่าแล้วอยากเขียนใหม่หมด ("โค้ดนี้เก่า/งง เขียนใหม่เร็วกว่า") — แต่การเขียนใหม่ = ทิ้งความรู้ที่สะสมมาหลายปี (bug fixes, edge cases, business logic ที่ไม่มีในเอกสาร) + เสี่ยงพังของที่ทำงานอยู่ · มือโปรอ่านให้เข้าใจก่อน แล้วแก้เฉพาะจุด

เปรียบเปรย: refactor = รีโนเวทบ้านเก่า · โครงสร้าง/ท่อ/สายไฟที่ยังดีก็เก็บไว้ (reuse) เปลี่ยนเฉพาะที่เก่าจริง (ห้องน้ำ สีผนัง) — ไม่มีใครทุบบ้านทั้งหลังเพราะอยากเปลี่ยนสีห้องนั่งเล่น · การ redesign Fujisawa ก็เหมือนกัน: เก็บ logic ที่ทำงานดี (qTranslate, เวลาทำการ, เมนู) เปลี่ยนเฉพาะ markup/หน้าตา

หลักการ 2: รู้จัก anatomy ของ theme ก่อนแตะ

ก่อนแก้บ้าน ต้องรู้ว่าห้องไหนอยู่ตรงไหน · theme WordPress มีโครงมาตรฐานที่ต้องอ่านออก:

โฟลเดอร์/ไฟล์หน้าที่ (เปรียบเปรยบ้าน)
header.php / footer.phpหัว-ท้ายที่ทุกหน้าใช้ร่วม (หลังคา + ฐานราก)
root: single-{cpt}.php, archive-{cpt}.phpห้องตามชนิด (บันได template hierarchy จาก Week 1)
templates/*.php (page templates)ห้องที่เลือกเองใน admin (ทางลัดข้ามบันได — Week 1)
template-parts/*.phpชิ้นส่วนที่ reuse ได้ (การ์ด, แบนเนอร์ — เฟอร์นิเจอร์ยกไปห้องไหนก็ได้)
inc/*.phpระบบหลังบ้าน (enqueue, helper, menu — ท่อ/สายไฟ)
functions.phpสารบัญที่ require ไฟล์ inc/ ทั้งหมด

หลักการ 3: เมื่อไหร่แทน markup (ก) / เมื่อไหร่ retrofit (ข)

docs/07 ให้ 2 ทางเลือก — เลือกต่อหน้าไม่ใช่ทั้งเว็บ:

(ก) แทนที่ markup(ข) retrofit
ทำอะไรรื้อ markup Bootstrap เดิม วาง markup + styles.css ดีไซน์ใหม่คงโครง Bootstrap เดิม ปรับ SCSS ให้ใกล้ดีไซน์
ข้อดีตรงดีไซน์ 100% · ได้ Unibar/hero/GEO ตามออกแบบเร็ว เสี่ยงพังน้อย
ข้อเสียงานเยอะ · ต้องถอด Bootstrap/SCSS เดิมดีไซน์อาจเพี้ยนจาก grid เดิม
ใช้กับหน้า GEO สำคัญ (หน้าแรก, single-whattosee, news)หน้ารองที่โครงใกล้เคียงอยู่แล้ว

หลักคิด: ไม่มีคำตอบเดียวถูก — dev ที่ดีเลือกเครื่องมือตามงาน · หน้าที่ธุรกิจแคร์ (GEO) ลงแรงทำให้ตรง หน้ารองประหยัดแรง

ส่วนที่ 1: ของเดิมที่ต้อง reuse (อย่าเขียนใหม่!)

docs/07 §4 ระบุชัดว่าอะไร reuse ได้ — การเขียนใหม่ = เสียเวลา + เสี่ยงพังของที่ทำงานดี:

ของเดิมทำอะไร · ทำไม reuse
qtranxf_use_language() · lang_label()ตัดภาษา 8 ภาษา (Week 5!) — logic ผ่านการใช้จริงมาแล้ว เขียนใหม่ = reinvent + เสี่ยงบั๊ก
template-parts/include-opentoday.phpมี logic คำนวณเวลาเปิด-ปิดครบแล้ว — Unibar (Week 6) เอามาใช้ต่อได้เลย ไม่ต้องเขียนใหม่
เมนู 4 ชุด (primary, other_menu, special_menu, sightseeing_menu)register ไว้แล้ว — แค่เรียก wp_nav_menu() ในที่ใหม่
thumb-medium 740×555 · set_pagesizeimage size + pagination ต่อหน้า (news=10, video=9) ตั้งไว้แล้ว
enqueue ผ่าน inc/scripts.phpโหลด CSS/JS ที่เดียว — Week 1 เรียนแล้วว่าห้าม hardcode ใน header
วิธีอ่านโค้ดคนอื่นให้เร็ว: ① เริ่มที่ functions.php ดูว่า require อะไรบ้าง (= สารบัญ) ② เปิด inc/scripts.php ดู asset ③ ไล่ template ตาม hierarchy ④ grep หา function name เวลาเจอสิ่งที่ไม่รู้จัก · อย่าอ่านทีละบรรทัดตั้งแต่ต้น — ไล่ตามเส้นทางข้อมูล (Week 1)

ส่วนที่ 2: 🤝 "class ในดีไซน์ = สัญญากับ CSS" (Risk R10)

เปรียบเปรย: ในดีไซน์ Fujisawa ไฟล์ styles.css เขียนกฎไว้ว่า .dt-hero { ... }, .answer-card { ... } — class เหล่านี้คือสัญญา: "ถ้า HTML มี class นี้ CSS จะแต่งหน้าให้แบบนี้" · ถ้า dev เปลี่ยนชื่อ class ใน markup = ผิดสัญญา = CSS หาไม่เจอ = ดีไซน์เพี้ยนทั้งหน้า

💥 Counter-example (Risk R10): เผลอเปลี่ยน class="dt-hero" เป็น class="hero-detail" เพราะคิดว่าชื่อสื่อกว่า → styles.css ที่อ้าง .dt-hero หาไม่เจอ → hero กลายเป็นกล่องเปล่าไม่มีสไตล์ · กฎเหล็ก: รักษา class เดิมที่ styles.css อ้างถึงไว้เป๊ะ — อยากเปลี่ยนดีไซน์ให้แก้ที่ตัวแปร :root ใน CSS ไม่ใช่ชื่อ class

โยงหลักการทอง 3 ข้อของโปรเจค: "ดีไซน์ HTML = พิมพ์เขียว — ใช้ styles.css เดิม ไม่ออกแบบใหม่ รักษา class เดิม" · งานของ dev คือห่อ markup ด้วย PHP ไม่ใช่แต่งหน้าใหม่

ส่วนที่ 3: อ่านโค้ดจริง — โครงของ single-whattosee.php

ไฟล์ dev-starter single-whattosee.php คือแม่แบบที่ Mission จะทำ · โครงของมัน = "รีโนเวท": เก็บ HTML/class จากดีไซน์ + เสียบ PHP ดึงข้อมูล:

<?php get_header(); ?>                            // 1

<?php while (have_posts()): the_post();          // 2
  $summary = get_field('summary_th');            // 3
  $hours   = get_field('opening_hours');
  $lat     = get_field('geo_lat');
  $lng     = get_field('geo_lng');
?>

  <div class="dt-hero">                           // 4 ← class จากดีไซน์ (สัญญา!)
    <h1 class="dt-title"><?php the_title(); ?></h1>  // 5

    <?php if ($hours): ?>                         // 6
      <div class="val"><?php echo esc_html($hours); ?></div>  // 7
    <?php endif; ?>

    <?php if ($lat && $lng): ?>                    // 8
      <a class="btn" href="https://maps.google.com/?q=<?php
         echo esc_attr($lat); ?>,<?php echo esc_attr($lng); ?>">  // 9
         ดูแผนที่</a>
    <?php endif; ?>
  </div>

<?php endwhile; ?>
<?php get_footer(); ?>                            // 10
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
1,10get_header()/get_footer() — ดึงหัว-ท้ายร่วม (รวม Unibar จาก Week 6) · ไม่ก๊อป markup header มาทุกหน้า (reuse)
2The Loop (Week 6) — main query ของหน้า single มีโพสต์เดียว แต่ยังต้องวน loop เพื่อให้ template tag ทำงาน
3ดึง ACF field ที่เพิ่มเองใน Week 4 — เก็บตัวแปรบนสุด อ่านง่าย + reuse
4class dt-hero จากดีไซน์เดิม — ห้ามเปลี่ยน (สัญญากับ CSS · ส่วนที่ 2)
5the_title() ผ่าน qTranslate filter อัตโนมัติ (Week 5) — แสดงภาษาตาม URL
6-7guard clause: มีค่าค่อยแสดง (Week 4) + esc_html() เสมอ
8-9จุดที่ dev-starter ยังละ escape ไว้ให้เติม! — $lat,$lng ต่อเข้า URL ต้อง esc_attr()/esc_url() · Mission จะหาจุดนี้แล้วเติม

🧹 Clean Code ประจำ Week 7 — refactor อย่างปลอดภัย

1. functions.php = สารบัญ ไม่ใช่โกดัง

❌ ยัดทุก logic ใน functions.php

// functions.php 1,500 บรรทัด
// enqueue + CPT + ajax + schema
// + reCAPTCHA (secret key ด้วย!)
// ปนกันหมด — หาอะไรไม่เจอ

✅ require ไฟล์ inc/ ตามหน้าที่

require 'inc/scripts.php';   // enqueue
require 'inc/template-tags.php';
require 'inc/ajax.php';
// secret key → wp-config.php constant
// แต่ละไฟล์ทำเรื่องเดียว

2. เก็บ class ดีไซน์ · เปลี่ยนหน้าตาที่ตัวแปร CSS

❌ เปลี่ยนชื่อ class = ผิดสัญญา CSS

<!-- เดิม .dt-hero -->
<div class="my-new-hero">
// styles.css หา .dt-hero ไม่เจอ
// → หน้าเพี้ยนทั้งบล็อก

✅ คง class · แก้สีที่ :root

<div class="dt-hero">  <!-- คงเดิม -->
/* styles.css */
:root { --wis: #15679f; }
// เปลี่ยนธีมสีที่ตัวแปรที่เดียว
กฎทอง: refactor ที่ดี = "พฤติกรรมเดิม โครงสร้างดีขึ้น" — ถ้าหน้าตา/การทำงานเปลี่ยนไปจากที่ควร แสดงว่าไม่ใช่ refactor แล้ว แต่เป็นการแก้ที่อาจพลาด · แก้ทีละน้อย ทดสอบบ่อย (โยง Week 6: ทดสอบด้วยตาทุกครั้ง)

📝 Quiz — ทดสอบความเข้าใจ

ข้อ 1: เจอ theme เก่าที่ "โค้ดงง อยากเขียนใหม่หมด" — ทำไมมืออาชีพมักไม่ทำ?

เพราะโค้ดเก่าเก็บความรู้ที่ไม่มีในเอกสาร: bug fixes, edge cases, business logic ที่สะสมมาหลายปี · เขียนใหม่ = ทิ้งของพวกนี้ + เสี่ยงพังของที่ทำงานอยู่ + ใช้เวลานานกว่าที่คิดเสมอ · ทางที่ถูก: อ่านให้เข้าใจ → refactor เฉพาะจุด (รีโนเวท ไม่ทุบสร้างใหม่) · reuse ของที่ดีอยู่แล้ว (qTranslate, เวลาทำการ, เมนู)

ข้อ 2: ทำไมห้ามเปลี่ยนชื่อ class ที่ styles.css อ้างถึง แล้วถ้าอยากปรับดีไซน์ต้องทำที่ไหน?

class คือ "สัญญา" ระหว่าง HTML กับ CSS — เปลี่ยนชื่อ class ใน markup = CSS หากฎที่ตรงกับ class เดิมไม่เจอ = สไตล์หาย หน้าเพี้ยนทั้งบล็อก · อยากปรับดีไซน์ให้แก้ที่ตัวแปร :root ใน styles.css (เช่น --wis สีแบรนด์) — มีผลทั้งเว็บโดยไม่แตะ markup · หลักการโปรเจค: "ดีไซน์ = พิมพ์เขียว ใช้ styles.css เดิม"

ข้อ 3 (security/production): reCAPTCHA secret key hardcode ใน functions.php — ทำไมอันตราย และแก้ยังไงก่อน commit แรก?

secret key ในโค้ด = พอ push ขึ้น Git ทุกคนที่เห็น repo (รวม history) เห็น key → ใช้ปลอมเป็นเว็บเราเรียก reCAPTCHA API ได้ / ถ้า repo หลุด public = key รั่วถาวร (แม้ลบทีหลัง history ยังมี) · แก้: ย้ายไป wp-config.php constant หรือ env var (getenv('RECAPTCHA_SECRET')) ที่ไม่เข้า Git (อยู่ใน .gitignore) — ทำก่อน commit แรก เพราะ key ที่เคยเข้า history แล้วลบยาก (Risk R4)

ข้อ 4: หน้าไหนของ Fujisawa ควรใช้ (ก) แทน markup / หน้าไหนใช้ (ข) retrofit? เพราะอะไร?

(ก) แทน markup → หน้า GEO สำคัญ (หน้าแรก, single-whattosee, news) เพราะต้องคุมดีไซน์+schema ให้ตรงเป๊ะ (ธุรกิจแคร์ AI search) · (ข) retrofit → หน้ารองที่โครง Bootstrap เดิมใกล้ดีไซน์อยู่แล้ว (ประหยัดแรง เสี่ยงพังน้อย) · หลักคิด: เลือกเครื่องมือตามคุณค่าของงาน ไม่ใช่ใช้วิธีเดียวทั้งเว็บ
