1) แนวคิด dev — ทำไมต้องเข้าใจ mental model ก่อน syntax

💡 Dev Concept

คนที่ "ใช้ WordPress เป็น" กับคนที่ "เข้าใจ WordPress" ต่างกันตรงนี้: คนแรกจำว่าต้องคลิกอะไร/copy โค้ดไหน — พอเจอโจทย์ใหม่ที่ไม่เคยเจอจะไปต่อไม่ได้ คนที่สองรู้ว่าข้อมูลไหลจากไหนไปไหน — เจอโจทย์ใหม่ก็ไล่เส้นทางข้อมูลแล้วหาจุดเสียบโค้ดเองได้

WordPress ทั้งระบบ (รวมถึง welovefukushima และ discover-fujisawa ที่วีดูแล) ตั้งอยู่บน 3 เสา:

  1. Data: ทุกเนื้อหาคือแถวใน wp_posts (ต่างกันแค่ post_type)
  2. Logic: ทุกการต่อเติมทำผ่าน hook (ไม่แก้ของเดิม แต่"เสียบ"เพิ่ม)
  3. Presentation: WP เลือกไฟล์ template ให้อัตโนมัติตามกติกาชื่อไฟล์ (template hierarchy)

2) เสาที่ 1 — "ทุกอย่างคือ wp_posts"

💡 Dev Concept

WordPress ไม่สร้างตารางใหม่ทุกครั้งที่มีเนื้อหาชนิดใหม่ — มันใช้ตารางเดียวคือ wp_posts แล้วแยกชนิดด้วยคอลัมน์ post_type ส่วนข้อมูลเพิ่มเติม (ราคา, พิกัด, URL วิดีโอ) เก็บใน wp_postmeta แบบ key–value ผูกกับ post ด้วย post_id

แนวคิดนี้เรียกว่า EAV (Entity–Attribute–Value): ยืดหยุ่นสูงมาก (เพิ่มชนิดข้อมูล/ฟิลด์ใหม่ได้โดยไม่ต้อง ALTER TABLE) แลกกับ query ที่ซับซ้อนขึ้นเมื่อ filter ด้วย meta เยอะๆ — trade-off นี้คือเหตุผลเบื้องหลังหลายการตัดสินใจที่จะเจอใน Week 3 และ 5

🟢 ตัวอย่างจริง #1 — Discover FUJISAWA (เว็บที่วีดูแล)

จาก SQL dump จริง (fujisawaadm_discoverfujisawadb_20260407.sql) ตาราง wp_posts มี 2,261 แถว แยกตาม post_type:

post_type จำนวน มันคืออะไรใน admin
attachment 1,303 ทุกรูป/PDF ใน Media Library
revision 505 ประวัติการแก้ไขแต่ละครั้งที่กด Update
news 95 CPT "News" ที่เห็นในเมนู admin
nav_menu_item 41 แต่ละลิงก์ในเมนู ก็เป็น post!
page 36 หน้า Pages
ssws__shop, ssse__event, whattosee ฯลฯ ~120 CPT อีก ~18 ชนิดของเว็บ

สังเกต: เมนู admin ที่วีเห็น (Tourism spots, Local Cuisine, Where to Stay...) ทุกอันคือตารางเดียวกัน ต่างแค่ค่าในคอลัมน์เดียว

🟢 ตัวอย่างจริง #2 — อุตสาหกรรม

WooCommerce (ระบบ e-commerce ที่ใช้กันมากที่สุดใน WP) ก็ใช้หลักเดียวกัน: สินค้า = post_type 'product', ออเดอร์ = post_type 'shop_order', ราคา = wp_postmeta key _price — เข้าใจหลักนี้ที่เดียว อ่านออกทั้ง ecosystem

🟠 Why — ทำไมต้องออกแบบแบบนี้ (และถ้าไม่... จะเป็นยังไง)

ทำไม WP ใช้ตารางเดียว? เพราะ WP เป็น CMS ที่ไม่รู้ล่วงหน้าว่าผู้ใช้จะมีเนื้อหาชนิดไหน ถ้าทุกเว็บต้อง CREATE TABLE เอง ปุ่ม "Add New Post Type" ใน CPT UI จะไม่มีทางทำงานได้ทันที

Counter-example: ทีม fujisawa เคยตัดสินใจตรงข้ามให้ดูแล้ว — ตาราง custom wp_fjs_contact, wp_import_* 18 ตารางใน DB เดียวกัน คือกรณีที่ dev เลือกสร้างตารางเองเพราะข้อมูล (ฟอร์ม contact, ข้อมูล migrate) ไม่ใช่ "เนื้อหาที่บรรณาธิการแก้" — เส้นแบ่งคือ: ถ้าคนทำคอนเทนต์ต้องแก้ผ่าน admin → ใช้ wp_posts, ถ้าเป็นข้อมูลระบบ → ตารางเอง วีจะเจอการตัดสินใจนี้เองใน Week 3

3) เสาที่ 2 — Hook = ปลั๊กไฟ

💡 Dev Concept

ทุกครั้งที่ WP โหลดหน้า มันวิ่งผ่านลำดับเหตุการณ์เดิมๆ (โหลด plugin → init → query → render → ปิดท้าย) และระหว่างทางมันตะโกนบอกจุดสำคัญผ่าน do_action('ชื่อจุด') — เหมือนรูปลั๊กไฟที่เปิดรอไว้ เราเอาโค้ดไปเสียบด้วย add_action() โดยไม่ต้องแก้ไฟล์ของ WP หรือ theme หลักเลย

หลักการ dev ที่อยู่เบื้องหลังคือ Open/Closed Principle: ระบบควร "ปิดรับการแก้ไข แต่เปิดรับการต่อขยาย" — WP core และ parent theme คือของที่จะถูกอัปเดตทับตลอดเวลา การแก้มันตรงๆ = งานหายเมื่ออัปเดต

Hook มี 2 ชนิด: action (ทำอะไรบางอย่าง ณ จุดนั้น — ไม่ต้องคืนค่า) และ filter (รับค่าไปแปลงแล้วต้องคืนค่ากลับเสมอ)

🟢 ตัวอย่างจริง #1 — welovefukushima (โค้ดที่รันอยู่จริง)

flavor-flavors-child/inc/schema.php เสียบ Schema.org markup เข้า <head> ของเฉพาะหน้า sightseeing เพื่องาน GEO/AEO โดยไม่แตะ Blocksy parent เลย:

// โค้ดจริง (ตัดมาบางส่วน) จาก inc/schema.php
function wlf_sightseeing_schema() {
    if ( ! is_singular( 'sightseeing' ) ) {
        return;  // ← ทำไมต้องมี: hook นี้ยิงทุกหน้า ถ้าไม่เช็คก่อน
    }            //   schema ท่องเที่ยวจะโผล่ในหน้า contact ด้วย = ผิดความหมาย
    // ... สร้าง JSON-LD TouristAttraction ...
}
add_action( 'wp_head', 'wlf_sightseeing_schema' );
// ↑ ทำไมเสียบ wp_head: เพราะ schema ต้องอยู่ใน <head> และ wp_head
//   คือรูปลั๊กเดียวที่ theme ทุกตัว "ต้อง" เปิดไว้ตรงนั้นตามกติกา WP
🟢 ตัวอย่างจริง #2 — filter ในงานจริง
// ตัวอย่าง filter: เปลี่ยนคำว่า "Read more" ทั้งเว็บโดยไม่ไล่แก้ template
add_filter( 'excerpt_more', function( $more ) {
    return ' …อ่านต่อ';   // ← filter "ต้องคืนค่า" เสมอ
});                          //   ถ้าลืม return → excerpt ทั้งเว็บกลายเป็นค่าว่าง!

นี่คือความต่าง action vs filter ที่เจอบ่อยสุดในสนาม: filter ลืม return = ข้อมูลหาย เพราะ WP เอาค่าที่คืน (null) ไปใช้ต่อจริงๆ

🟠 Why — ทำไมห้ามแก้ core/parent theme ตรงๆ

Counter-example คลาสสิก: สมมุติแก้ blocksy/header.php ตรงๆ เพื่อเพิ่ม banner → วันที่ Blocksy ออกอัปเดต (ซึ่ง parent theme อัปเดตบ่อยมาก) ไฟล์ถูกทับ งานหายทั้งหมดแบบไม่มี warning — นี่คือเหตุผลเดียวที่ flavor-flavors-child ต้องเป็น child theme และทุก logic เสียบผ่าน hook

เหตุผลเดียวกันนี้จะตอบคำถาม Week 2 ว่าทำไม CPT ควรอยู่ใน plugin: โค้ดต้องอยู่รอดเมื่อสิ่งที่มันเกาะอยู่ถูกอัปเดตหรือเปลี่ยน

4) เสาที่ 3 — Template Hierarchy: บันไดเลือกไฟล์

💡 Dev Concept

WP ไม่มี config ผูก URL กับไฟล์ — มันใช้ convention over configuration: ดูว่า request คืออะไร (archive ของ CPT ไหน? single post? taxonomy ไหน?) แล้วไล่หาไฟล์ตามลำดับชื่อ เจอไฟล์แรกที่มีอยู่จริง = ใช้ไฟล์นั้น ถ้าไม่เจอเลยจะจบที่ index.php เสมอ

นี่คือเหตุผลที่ "แค่ตั้งชื่อไฟล์ถูก ก็ได้หน้าใหม่ฟรี" — และเหตุผลที่ตั้งชื่อผิดตัวเดียวแล้วงงว่าทำไมหน้าไม่เปลี่ยน

🟢 ตัวอย่างจริง #1 — ไฟล์ที่มีอยู่จริงใน flavor-flavors-child
Request ไฟล์ที่ WP เลือก (มีจริงใน theme)
หน้ารวม sightseeing archive-sightseeing.php
หน้ารายละเอียดสถานที่ single-sightseeing.php
หน้ารวม blog / event archive-bloglist.php / archive-eventcalendar.php
หน้า Pages เฉพาะ page-about.php, page-foods.php, page-spring.php ... (slug ตรงชื่อไฟล์)
หน้าแรก front-page.php (ชนะทุกอย่างเมื่อตั้ง static front page)

💡 หน้าจอ CPT UI → Registered Types/Taxes ที่วีเปิดดู มีคอลัมน์ "Template Hierarchy" บอกบันไดนี้ของทุก CPT ตรงๆ — ใช้เป็น cheat sheet ได้เลยตอนแตะ theme จริงใน Week 7

🟢 ตัวอย่างจริง #2 — theme fujisawa จริงใช้ "Page Template" (ทางลัดข้ามบันได)

เปิด theme fujisawa บน production จะเจอความจริงที่หักมุม: หน้าแรกไม่ใช่ front-page.php — มันคือ Page ชื่อ "Discover FUJISAWA" ที่เลือก Page Template templates/home.php จาก dropdown ใน Page Attributes

Page template คือไฟล์ที่ประกาศ comment Template Name: Home ไว้หัวไฟล์ → WP เอาไปโชว์ให้เลือกใน admin → ถ้าถูกเลือก มันชนะบันไดของ page ทั้งหมด — เพราะ "คนเลือกเอง" ย่อมชนะ "convention"

theme นี้มี templates/ ถึง 20 ไฟล์ (about.php, news.php, cuisine.php...) = สไตล์ "page template เป็นหลัก" ต่างจาก wlf ที่ใช้ hierarchy เป็นหลัก — งานจริงเจอได้ทั้ง 2 สไตล์ ต้องอ่านออกทั้งคู่ (เจาะลึกใน Week 7)

อุตสาหกรรม: WooCommerce ใช้บันไดเดียวกันทำ template override — copy ไฟล์จาก plugin มาไว้ {theme}/woocommerce/ แล้วไฟล์ใน theme ชนะของ plugin ทันที หลักการเดียวกัน: ไฟล์ที่เจาะจงกว่า/ใกล้ตัวกว่า ชนะ

🟠 Why — ทำไม convention ดีกว่า config (ในเคสนี้)

เทียบกับ framework ที่ต้องประกาศ route เอง (Laravel: Route::get('/spot/{slug}', ...)): WP เลือกทางที่คนไม่เขียนโค้ดก็เพิ่มหน้าได้ (สร้าง page ใน admin → ได้ URL ทันที) — เหมาะกับ CMS แต่แลกกับความ implicit: ไม่มีที่เดียวที่บอกว่า URL ไหนใช้ไฟล์ไหน dev ใหม่เลยงง วิธีแก้ของมือโปร: จำบันไดไม่กี่ขั้น + ใช้ปลั๊กอินช่วย debug (เช่น Query Monitor บอกว่าหน้านี้ใช้ template ไหน) — จะติดตั้งใน sandbox สัปดาห์นี้

5) 🎯 Mission 1 — repo wp-hooks-lab

🤖 กฎ AI สองโหมด: Mission นี้ปิด AI เขียนมือเท่านั้น — ติดเกิน 20 นาที ส่งข้อความมาในแชท ผมให้ hint ทีละชั้น (ไม่เขียนแทน) เปิด AI ได้เฉพาะตอนอ่านทบทวนเนื้อหานอก Mission
เป้าหมาย

สร้าง plugin ตัวแรกของวีที่ใช้ hook 3 จุด — เล็กพอเขียนจบใน 2-3 ชม. แต่ครบเครื่องพอเป็นผลงาน GitHub ชิ้นแรกของแทร็ก

1
เตรียม sandbox (= Phase 0 ของงาน redesign จริง) — ติดตั้ง WP ใหม่ใน XAMPP (เช่น htdocs/wp-sandbox) แยกจากเว็บจริงเด็ดขาด · ให้เวอร์ชันใกล้ production ที่สุด (PHP 8.2 · WordPress 7.0 · MariaDB 10.6 — อ้างอิง docs/08 §1.1 ของโปรเจค V2.1) + ติดตั้งปลั๊กอิน Query Monitor ไว้ debug — sandbox ตัวนี้จะใช้ต่อยอดยาวถึง Week 9
2
สร้าง plugin — โฟลเดอร์ wp-content/plugins/wp-hooks-lab/ มีไฟล์หลัก wp-hooks-lab.php พร้อม plugin header ที่ถูกต้อง และกัน direct access (if ( ! defined( 'ABSPATH' ) ) exit; — ไปดูว่าทำไมทุกไฟล์ของ wlf มีบรรทัดนี้)
3
เสียบ hook 3 จุด (โจทย์บังคับ):
① action บน wp_head — เพิ่ม meta tag <meta name="developer" content="..."> เฉพาะหน้า front page เท่านั้น (ต้องเช็ค conditional tag เอง)
② filter บน the_title — เติม 🔥 หน้า title เฉพาะ post ที่เผยแพร่เกิน 30 วัน (ฝึกรับค่า-แปลง-คืนค่า + ใช้ข้อมูลของ post)
③ filter บน excerpt_length — เปลี่ยนความยาว excerpt เป็น 30 คำ
4
README มาตรฐาน portfolio — อธิบายว่า plugin ทำอะไร, ตาราง hook ที่ใช้ (hook ไหน / action หรือ filter / ทำไมเลือกจุดนั้น), วิธีติดตั้ง, และ AI Usage section (ระบุว่าเขียนมือ 100%)
5
Push ขึ้น GitHub — commit แยกเป็นขั้น (feat: add plugin skeleton, feat: add front-page meta tag action, ...) ตาม portfolio-standard ของแทร็กหลัก
✅ Definition of Done
  • Plugin activate ได้ไม่มี error / ทำงานครบ 3 ข้อ (ดูผลใน view-source + หน้า blog)
  • ลอง ลืม return ใน filter ② ดูหนึ่งครั้งแล้วดูผลที่เกิด → เขียนสิ่งที่เห็นใส่ README (บั๊กนี้ต้องเคยเจอกับตาตัวเอง)
  • เปิด plugin แล้วไปดู เมนูเว็บ + หน้ารายการ post ใน admin — 🔥 โผล่ในที่ที่ไม่ได้ตั้งใจไหม? เพราะอะไร? → เขียนสิ่งที่เจอ + วิธีกันใส่ README (คำใบ้: the_title ยิงทั้งใน admin และกับ nav menu item — ดู is_admin(), in_the_loop()) — กับดักจริงอันดับ 2 ของ filter รองจากลืม return
  • README ครบตามข้อ 4 + commit history อ่านรู้เรื่อง
  • ตอบ Quiz ด้านล่างได้โดยไม่เปิดเนื้อหา

6) คีย์เวิร์ดภาพจำ + Quiz

🔑 คีย์เวิร์ดภาพจำ (ทวนทุกศุกร์ — Friday Rebuild)
"ทุกอย่างคือ wp_posts" "field ลงเอยที่ postmeta" "hook = ปลั๊กไฟ" "filter ต้องคืนค่าเสมอ" "แก้ parent = งานหายตอนอัปเดต" "template hierarchy = บันไดเลือกไฟล์" "Page Template = ทางลัดข้ามบันได" "ไม่เจอไฟล์ไหนเลย → index.php"
📝 Quiz ความเข้าใจ (ตอบเองก่อน แล้วค่อยกางเฉลย — ส่งคำตอบในแชทให้ผมตรวจได้)

Q1. ใน SQL dump ของ discover-fujisawa มี revision 505 แถวใน wp_posts — revision คืออะไร และทำไม WP เก็บมันเป็น "post"?

เฉลยประวัติการแก้ไขแต่ละครั้ง — WP เก็บเป็นแถวใน wp_posts (post_type=revision, post_parent ชี้ตัวจริง) เพราะหลัก "ทุกอย่างคือ wp_posts": ไม่ต้องมีตารางใหม่ ใช้กลไก query/restore เดิมได้หมด

Q2. ต่างกันยังไง: add_action กับ add_filter — และอันไหน "ลืม return แล้วข้อมูลหาย"?

เฉลยaction = ทำงาน ณ จุดนั้น ไม่ต้องคืนค่า / filter = รับค่า→แปลง→ต้องคืนค่า เพราะ WP เอาค่าที่คืนไปใช้ต่อ ลืม return = คืน null = ข้อมูลส่วนนั้นหายทั้งเว็บ

Q3. ทำไม welovefukushima ต้องเป็น child theme ของ Blocksy แทนที่จะแก้ Blocksy ตรงๆ?

เฉลยBlocksy ถูกอัปเดตจากผู้พัฒนาเสมอ การแก้ตรงๆ จะถูกไฟล์ใหม่ทับหายหมด child theme + hook ทำให้ของเราอยู่คนละที่กับของที่ถูกอัปเดต (Open/Closed Principle)

Q4. URL /sightseeing/oze-park/ — WP จะลองหาไฟล์อะไรบ้าง เรียงลำดับ? และถ้าไม่มีสักไฟล์?

เฉลยsingle-sightseeing-oze-park.php → single-sightseeing.php → single.php → singular.php → index.php (ปลายทางสุดท้ายเสมอ)

Q5. ข้อมูล "เวลาเปิด-ปิด" ของสถานที่ท่องเที่ยวที่กรอกผ่าน ACF ไปอยู่ตารางไหนใน DB?

เฉลยwp_postmeta (key-value ผูกกับ post_id) — ACF คือ UI ที่สวยขึ้นของ postmeta ไม่ใช่ที่เก็บข้อมูลชนิดใหม่

Q6. ทีม fujisawa สร้างตาราง wp_fjs_contact เองแทนที่จะใช้ wp_posts — เกณฑ์ตัดสินใจคืออะไร?

เฉลยข้อมูลที่บรรณาธิการแก้ผ่าน admin → wp_posts / ข้อมูลระบบ (log ฟอร์ม, ข้อมูล import) ที่ไม่ใช่ "เนื้อหา" → ตารางเอง ได้ schema ตรงไปตรงมาและ query เร็วกว่า

Q7. หน้าแรกของ discover-fujisawa ใช้ templates/home.php ทั้งที่บทเรียนบอกว่า front-page.php "ชนะทุกอย่าง" — เป็นไปได้ยังไง?

เฉลยfront-page.php ชนะก็ต่อเมื่อมีไฟล์นั้นอยู่จริง — theme fujisawa ไม่มีไฟล์นี้ WP จึงไล่บันไดต่อ และหน้าแรกถูกตั้งเป็น static Page ที่เลือก Page Template "Home" ใน Page Attributes ซึ่ง page template ที่คนเลือกเองชนะ page-{slug}.php / page.php ในบันไดของ page — สรุป: บันไดทำงานเสมอ แต่ "ทางลัดที่คนเลือก" แทรกได้ และไฟล์ที่ไม่มีอยู่จริงข้ามไปเฉยๆ
