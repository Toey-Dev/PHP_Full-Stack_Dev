🛡 Security + Deploy — ปิดแทร็ก

สัปดาห์สุดท้าย รวมทุกอย่างเข้าด้วยกัน — security ครบวงจร (escape ที่ถักมาตั้งแต่ Week 4 + ขาเข้า) และวงจร deploy มืออาชีพ · จุดนี้คือที่ที่จุดแข็ง pentest ของวีมาบรรจบกับโค้ด WP ที่เขียนเอง ผ่าน case study จริง 2 ชิ้น — กลายเป็น "dev ที่ audit ระบบตัวเองได้" ซึ่งเป็นจุดขายใน portfolio

⏱ เนื้อหา ~3 ชม. + Mission ~4 ชม. · 🎯 Mission (Phase 4): SECURITY.md audit checklist + .gitlab-ci.yml lint

🔑 คีย์เวิร์ดภาพจำ

🚪 อย่าเชื่อ input (sanitize ขาเข้า) 🛡 escape ทุก output (ขาออก — Week 4) 🎫 nonce = ตั๋วกันปลอมคำขอ (CSRF) ☠️ uploads ต้องรัน PHP ไม่ได้ (webshell) 🔑 secret ไม่เข้า Git (env/wp-config) 🚀 deploy แตะแค่ theme = rollback ง่าย

🧠 แนวคิด & หลักการ

หลักการ 1: กฎ 2 ข้อที่สรุป security ฝั่ง dev ได้ 80%

วีมาจากสาย pentest — เข้าใจว่า "ช่องโหว่มาจากไหน" อยู่แล้ว · บทนี้เชื่อมมุมนั้นเข้ากับโค้ดที่เขียนเอง · สองกฎที่ครอบเกือบทุกช่องโหว่:

กฎความหมายฟังก์ชัน WP
🚪 อย่าเชื่อ input (ขาเข้า)ทุกอย่างที่มาจากผู้ใช้ (form, URL, API) อาจมีพิษ — ทำความสะอาดก่อนใช้sanitize_text_field(), absint(), check_ajax_referer()
🛡 escape ทุก output (ขาออก)ก่อนแสดงอะไรออกจอ แปลงให้ไร้พิษ (Week 4 ถักมาแล้ว)esc_html(), esc_url(), esc_attr()

เปรียบเปรย: ขาเข้า = ด่านตรวจคนเข้าอาคาร (สแกนกระเป๋าก่อนเข้า) · ขาออก = ด่านตรวจก่อนขึ้นเวที (เช็คอีกทีก่อนออกสู่สาธารณะ) · ระบบที่ปลอดภัยมีทั้งสองด่าน — เชื่อว่า input สะอาดแล้วเลยไม่ escape ตอน output = พลาด (เพราะ input อาจผ่านมาหลายทาง)

หลักการ 2: Defense in depth — หลายชั้นดีกว่าชั้นเดียว

ไม่มีด่านไหนกันได้ 100% — จึงวางหลายชั้น · ถ้าชั้นหนึ่งพลาด ยังมีชั้นถัดไป · เช่น ฟอร์ม AJAX: nonce (กัน CSRF) → sanitize (ล้าง input) → capability check (เช็คสิทธิ์) → escape (ตอนแสดงผลกลับ)

หลักการ 3: security ไม่ใช่ "งานตอนท้าย" — ถักตลอดทาง

โยงทั้งแทร็ก: เราไม่รอมาทำ security สัปดาห์สุดท้าย — escape ถักตั้งแต่ Week 4, nonce/sanitize จะใช้ทุกครั้งที่มี form, show_in_rest คิดตั้งแต่ Week 2 · บทนี้แค่รวบยอด + เพิ่มมุมที่ยังขาด (ขาเข้า/uploads/deploy) + ทำ audit checklist · หลักคิด: security เป็นนิสัย ไม่ใช่ขั้นตอน

ส่วนที่ 1: ขาเข้า — sanitize + nonce (เก็บที่ยังขาด)

Week 4 ทำ escape ขาออกไปแล้ว · คราวนี้เก็บขาเข้า — สำคัญเมื่อมีฟอร์ม/AJAX (เช่น filter สถานที่, ฟอร์มติดต่อ):

add_action('wp_ajax_filter_spots', 'df_filter_spots');       // 1
add_action('wp_ajax_nopriv_filter_spots', 'df_filter_spots');
function df_filter_spots() {
  check_ajax_referer('spot_filter', 'nonce');                // 2
  $area  = sanitize_text_field($_POST['area'] ?? '');        // 3
  $page  = absint($_POST['page'] ?? 1);                      // 4
  $q = new WP_Query([
    'post_type' => 'whattosee',
    'paged'     => $page,
    'tax_query' => [['taxonomy'=>'spot_cat','terms'=>$area,'field'=>'slug']], // 5
  ]);
  // ...loop + echo esc_html(...)...                          // 6
  wp_die();                                                   // 7
}
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
12 hook: wp_ajax_ (คน login) + wp_ajax_nopriv_ (คนทั่วไป) — เว็บ public ต้องมีตัวหลัง ไม่งั้น AJAX ใช้ได้แค่คน login
2🎫 nonce ก่อนทำอะไรทั้งสิ้น — check_ajax_referer() เช็ค "ตั๋ว" ที่แนบมา · กัน CSRF (เว็บอื่นหลอกส่งคำขอในนามผู้ใช้) · ผิด/ไม่มีตั๋ว = หยุดทันที
3sanitize_text_field() ล้าง $_POST['area'] — ตัด HTML/ช่องว่างเกิน/บรรทัดใหม่ · ?? '' กัน undefined index (PHP 8)
4absint() บังคับเป็นจำนวนเต็มบวก — เลขหน้าต้องเป็นตัวเลข ไม่ใช่ "1; DROP TABLE" · แปลง type = กันพิษไปในตัว
5ใช้ค่าที่ sanitize แล้วใน tax_query · WP_Query เตรียม SQL แบบ prepared ให้ = กัน SQL injection อีกชั้น (ไม่ต้องต่อ SQL เอง)
6ขาออกยัง escape เสมอ (Week 4) — input ผ่าน sanitize แล้วก็ยัง escape ตอน echo (defense in depth)
7wp_die() ปิดท้าย AJAX handler — ไม่งั้น WP ต่อ output "0" มาท้าย response
เปรียบเปรย nonce: = ตั๋วคอนเสิร์ตที่มีเลขกำกับ + หมดอายุ · เว็บอื่นหลอกให้ผู้ใช้กดส่งฟอร์มได้ แต่ไม่มีตั๋วที่ถูกต้อง = เข้าไม่ได้ · ต่างจากรหัสผ่าน (nonce ใช้ครั้งเดียว/ช่วงเวลาสั้น) — กันการ"ปลอมคำขอ" โดยเฉพาะ

ส่วนที่ 2: 🕵️ Case Study จริง #1 — Webshell ใน uploads

พบไฟล์ลักษณะ webshell 2 ไฟล์ใน backup uploads ของ discover-fujisawa (uploads/2019/n99bq6bc.php, uploads/2021/leca126n.php — โค้ด obfuscated) · นี่คือจุดที่มุม pentest ของวีวิเคราะห์ได้ลึกกว่า dev ทั่วไป:

คำถามสืบสวนวิเคราะห์
webshell คืออะไรไฟล์ PHP ที่แฮกเกอร์อัปโหลดเข้าเซิร์ฟเวอร์ = "ประตูหลัง" ให้รันคำสั่งบนเครื่องได้จากระยะไกล (อ่านไฟล์, แก้ DB, ส่ง spam)
เข้ามาทางไหนช่องอัปโหลดที่ไม่ validate ชนิดไฟล์ (เผลอรับ .php), ปลั๊กอิน/theme เก่ามีช่องโหว่, หรือบัญชี admin ถูกเดารหัส
ทำไม PHP ใน uploads ถึงรันได้โฟลเดอร์ uploads/ ควรเก็บแค่รูป/PDF — แต่ถ้า server ตั้งค่าให้รัน .php ทุกที่ ไฟล์ที่หลุดเข้าไปก็รันได้ · นี่คือช่องที่ปิดได้ที่ระดับ server
🔒 วิธีปิด (ระดับ Apache/nginx) — สำคัญที่สุด: บล็อกการรัน PHP ในโฟลเดอร์ uploads ทั้งหมด · แม้แฮกเกอร์อัปโหลด webshell เข้าไปได้ ก็รันไม่ได้ (กลายเป็นไฟล์ข้อความเฉยๆ) 
# .htaccess ใน wp-content/uploads/ (Apache)
<FilesMatch "\.php$">
  Require all denied
</FilesMatch>
+ action item จริง (README ระบุ): ต้องตรวจ server จริง + แจ้งทีมก่อน restore uploads ใดๆ — เรื่องนี้ไม่รอหลักสูตร

หลักคิด: "assume breach" (สมมุติว่าถูกเจาะได้) — ไม่พึ่งแค่กันไม่ให้อัปโหลด (ด่านเดียว) แต่วางด่านที่ 2: ต่อให้อัปโหลดได้ก็รันไม่ได้ · defense in depth ในโลกจริง

ส่วนที่ 3: 🔑 Case Study จริง #2 — Secret hardcode (Risk R4)

ในธีมจริงมี reCAPTCHA v3 secret key เขียนตายใน functions.php (Week 7 เตือนไว้) — วิเคราะห์เชิง security:

ประเด็นทำไมอันตราย
key อยู่ในโค้ดทุกคนที่เห็น repo (รวม Git history) เห็น key · แม้ commit ลบทีหลัง — history ยังเก็บไว้ถาวร (ต้อง rewrite history + revoke key)
ถ้า repo หลุด publickey รั่วถาวร — แฮกเกอร์ปลอมเป็นเว็บเราเรียก API, หรือทำ reCAPTCHA ของเราใช้ไม่ได้
ตรวจเจอยังไงเครื่องมือ secret scanner (gitleaks, GitHub secret scanning) จับ pattern ของ key อัตโนมัติ — บางทีแฮกเกอร์สแกน repo หา key เร็วกว่าเรารู้ตัว
วิธีแก้ (ทำก่อน commit แรก): ย้าย key ไป wp-config.php constant หรือ environment variable ที่อยู่ใน .gitignore: 
// wp-config.php (ไม่เข้า Git)
define('RECAPTCHA_SECRET', 'xxx');
// functions.php
$secret = defined('RECAPTCHA_SECRET') ? RECAPTCHA_SECRET : getenv('RECAPTCHA_SECRET');

ส่วนที่ 4: 🚀 วงจร Deploy มืออาชีพ (docs/08)

เปรียบเปรย: deploy เว็บ production ที่มีเนื้อหาจริง = ผ่าตัดคนไข้ที่ยังตื่นอยู่ · ต้องแม่นยำ ย้อนกลับได้ ไม่แตะอวัยวะที่ไม่เกี่ยว · หลักเหล็ก: deploy = sync เฉพาะ themes/fujisawa/ ไม่แตะ DB/uploads/plugins

ขั้นทำอะไร · ทำไม
1. MR + review1 template = 1 branch = 1 MR — เล็ก review ง่าย ย้อนง่าย · ทีมเห็น diff ก่อน merge
2. CI lintphp -l (syntax) + phpcs (WordPress Coding Standards) — จับ error ก่อนขึ้น prod
3. rsync เฉพาะ themersync --delete themes/fujisawa/ — sync แค่โฟลเดอร์ theme · --exclude scss/node_modules · ไม่แตะ DB/uploads
4. post-deployflush permalink (ถ้ามี slug ใหม่) · ล้าง cache · ping Search Console · smoke test
5. rollbackgit revert + deploy ซ้ำ — DB ไม่กระทบเพราะไม่เคยแตะ = ปลอดภัย
ทำไม "แตะแค่ theme" สำคัญ: production มีเนื้อหา 2,000+ ชิ้น + uploads + DB จริง · ถ้า deploy เผลอ sync ทับ DB/uploads = ข้อมูลลูกค้าเสียหายกู้ยาก · จำกัดขอบเขตให้แตะแค่ "โค้ด theme" ทำให้ทุก deploy ปลอดภัย + rollback ง่าย (revert แล้ว deploy ใหม่ ข้อมูลไม่กระทบ)

ตัวอย่างประยุกต์

🏢 งานวี: จุดขายใน portfolio ของวีคือ "dev ที่ audit ระบบตัวเองได้" — น้อยคนที่ทั้งเขียน WP ได้และมองเห็นช่องโหว่แบบ pentester · การวิเคราะห์ webshell + secret ในเว็บจริงที่ดูแลเอง แล้วเขียน SECURITY.md แก้ = หลักฐานทักษะที่หายากในตลาด

🌏 อุตสาหกรรม: ทั้ง 2 กรณีอยู่ใน OWASP Top 10: webshell = A03 Injection/unrestricted upload · secret hardcode = A07 (identification/auth failures) + secret management · ทุกบริษัทที่ทำ DevSecOps มี secret scanning + upload restriction ใน pipeline · Samy worm, การเจาะผ่าน uploads เป็นเคสคลาสสิกที่สอนกันในทุกคอร์ส security

🧹 Clean Code ประจำ Week 9 — เขียนโค้ดที่ปลอดภัยเป็นค่าเริ่มต้น

1. nonce + sanitize ทุก handler (ไม่มีข้อยกเว้น)

❌ ใช้ $_POST ตรงๆ — ช่องโหว่

function df_filter() {
  $area = $_POST['area'];   // ดิบ!
  $q = new WP_Query([...$area...]);
  // ไม่มี nonce · ไม่ sanitize
}

✅ ตั๋ว + ล้าง ก่อนใช้เสมอ

function df_filter() {
  check_ajax_referer('spot_filter','nonce');
  $area = sanitize_text_field(
            $_POST['area'] ?? '');
  // ปลอดภัยแล้วค่อยใช้
}

2. secret ผ่าน constant/env — ไม่เขียนในโค้ด

❌ key ในโค้ด = รั่วเข้า Git

$secret = '6Lc9x...ตัวจริง...';
// push = ทุกคนเห็น + history ถาวร

✅ อ่านจากที่ไม่เข้า Git

$secret = getenv('RECAPTCHA_SECRET');
// หรือ wp-config constant
// อยู่ใน .gitignore
กฎทอง: "secure by default" — เขียนแบบปลอดภัยตั้งแต่แรกจนเป็นนิสัย ไม่ใช่ "ค่อยมา harden ทีหลัง" · เพราะช่องโหว่ที่เผลอเขียนไว้ มักถูกลืมจนแฮกเกอร์เจอก่อน (โยงทั้งแทร็ก: escape ตั้งแต่ W4, nonce ทุก form, secret นอกโค้ด)

📝 Quiz — ทดสอบความเข้าใจ (ข้อสุดท้ายของแทร็ก)

ข้อ 1: ต่างระหว่าง sanitize (ขาเข้า) กับ escape (ขาออก) — ทำไมต้องมีทั้งคู่ ไม่พึ่งอันเดียว?

sanitize = ล้าง input ตอนรับเข้า (ตัดสิ่งไม่ต้องการก่อนเก็บ/ใช้) · escape = แปลงให้ปลอดภัยตอนแสดงออกจอ (ตามบริบท html/url/attr) · ต้องมีทั้งคู่เพราะ: ① input อาจเข้ามาหลายทาง (บางทางไม่ผ่าน sanitize) ② ข้อมูลใน DB อาจมาจากยุคก่อนที่ยังไม่ sanitize ③ defense in depth — ด่านหนึ่งพลาดยังมีอีกด่าน · escape ตอน output คือด่านสุดท้ายที่ขาดไม่ได้

ข้อ 2: nonce กัน CSRF ยังไง? ต่างจากรหัสผ่านตรงไหน?

nonce = โทเคนสุ่มที่ผูกกับ action + ผู้ใช้ + ช่วงเวลา · เว็บอื่นที่พยายามหลอกให้ผู้ใช้ส่งคำขอ (CSRF) ไม่มี nonce ที่ถูกต้อง = ถูกปฏิเสธ · ต่างจากรหัสผ่าน: รหัสผ่านพิสูจน์ "คุณเป็นใคร" (ใช้ซ้ำได้นาน) · nonce พิสูจน์ "คำขอนี้มาจากฟอร์มจริงของเรา ไม่ใช่เว็บปลอม" (ใช้ครั้งเดียว/สั้น) — คนละหน้าที่

ข้อ 3 (security/production): ทำไม "block PHP execution ใน uploads" สำคัญกว่าแค่ validate ตอนอัปโหลด?

validate ตอนอัปโหลดคือด่านเดียว — ถ้าพลาด (ช่องโหว่ในปลั๊กอิน, bypass การเช็คชนิดไฟล์) webshell หลุดเข้าไปได้ · แต่ถ้า server บล็อกการรัน .php ใน uploads ไว้ด้วย (ด่าน 2) ต่อให้ webshell หลุดเข้าไป ก็เป็นแค่ไฟล์ข้อความรันไม่ได้ = ไร้พิษ · หลัก "assume breach + defense in depth": สมมุติด่านแรกพลาดเสมอ แล้ววางด่านรับ · นี่คือเคสจริงจาก backup ของ discover-fujisawa

ข้อ 4: ทำไม deploy production ควรแตะแค่ themes/fujisawa/ ไม่แตะ DB/uploads? เชื่อมกับ rollback ยังไง?

production มีเนื้อหา/uploads/DB จริงที่ทีมคอนเทนต์ดูแล — ถ้า deploy ทับของพวกนี้ = ข้อมูลลูกค้าเสียหาย กู้ยาก · จำกัดให้แตะแค่ "โค้ด theme" ทำให้: ① deploy ปลอดภัย (ไม่มีทางแตะข้อมูล) ② rollback ง่าย — ถ้า theme ใหม่พัง แค่ git revert + deploy ซ้ำ DB ไม่เคยถูกแตะจึงไม่กระทบเลย · หลัก: จำกัด blast radius ของทุกการเปลี่ยนแปลง
