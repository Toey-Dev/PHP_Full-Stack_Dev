หลักการ 1: เว็บหลายภาษ มี 2 สถาปัตยกรรม — Fujisawa เลือกแบบ "รวมช่องเดียว"

เปรียบเปรย: qTranslate = กล่องที่มีลิ้นชักลับซ้อนกัน 8 ชั้น เปิดจากข้างนอกเห็นเป็นกล่องเดียว แต่มีล่าม (filter) ยืนอยู่คอยดึงเฉพาะลิ้นชักภาษาที่ผู้ชมขอออกมา · WPML = ตู้ 8 ใบเรียงกัน แต่ละใบคือภาษาหนึ่ง

หลักการ 2: ข้อมูลถูกเก็บ "รวม" แต่แสดง "แยก" ด้วย filter

นี่คือหัวใจ — ข้อมูลใน DB หน้าตาแบบนี้ (ช่องเดียว):

[:en]Enoshima Sea Candle[:th]ประภาคารเอโนชิมะ[:ja]江の島シーキャンドル[:]

แต่คนเปิดหน้า /th/ เห็นแค่ "ประภาคารเอโนชิมะ" — เพราะ qTranslate เสียบ filter (จำ Week 1: filter = รับค่ามาแปลงแล้วคืน) ที่ the_title, the_content ฯลฯ คอยตัดเฉพาะภาษาที่ URL ขอออกมาก่อนแสดง · เชื่อมกลับ Week 4: field ชนิด qtranslate_* ก็เก็บรูปแบบนี้ และ get_field() ผ่าน filter เดียวกัน

หลักการ 3: เมื่อไหร่ข้อมูลผ่าน filter อัตโนมัติ / เมื่อไหร่ต้องจัดการเอง

  • ผ่านอัตโนมัติ: the_title(), the_content(), the_field(), get_field() — qTranslate เสียบ filter ไว้ให้แล้ว
  • ต้องจัดการเอง: เวลาเอาข้อความหลายภาษาไปใช้นอกเส้นทางปกติ (เช่น สร้าง JSON, ส่ง JS, ตัวแปรที่ประกอบเอง) — ต้องเรียก qtranxf_use_language($lang, $text) ตัดภาษาเองก่อน

ส่วนที่ 2: 🐞 บั๊กจริงในธีม — lang_label() out-of-sync (Risk R8)

นี่คือบั๊กจริงที่เอกสารส่งมอบระบุไว้ว่า "รอ dev แก้" — และจะเป็น Mission ของวี · ต้นเหตุคือข้อมูล 2 แหล่งไม่ตรงกัน:

แหล่งภาษาที่มี
qTranslate UI จริง (ภาษา active)en · th · tw · cn · kr · fr · es · vn (8 ภาษา · มี vn)
helper lang_label() ในธีมมี ja (日本語) · แต่ไม่มี vn

เปรียบเปรย: เหมือนเมนูอาหารหน้าร้านเขียนว่า "มีเวียดนาม" แต่ในครัวไม่มีสูตรเวียดนาม กลับมีสูตรญี่ปุ่นที่เอาออกจากเมนูไปแล้ว — ลูกค้าเวียดนามสั่งแล้วได้จานเปล่า

💥 ผลของบั๊ก: เมื่อผู้ใช้เปิดภาษาเวียดนาม (/vn/) helper lang_label('vn') หาไม่เจอ → คืนค่าว่าง/ผิด → ปุ่มสลับภาษาแสดงเพี้ยน หรือ label หาย · และภาษา ja ที่ helper ยังมีอยู่ก็ไม่มีแท็บใน UI แล้ว = โค้ดตายที่รอลบ
วิธีแก้ (= Mission): เติม vn เข้า lang_label() + ยืนยันกับทีมว่า ja ยังต้องรองรับไหม (base ที่ไม่โชว์แท็บ หรือเลิกแล้ว) → ถ้าเลิกก็ลบออก

อ่านโค้ด helper ทีละส่วน (สมมุติโครงตามที่เอกสารบรรยาย)

function lang_label( $lang ) {                 // 1
    $labels = [                                 // 2
        'en' => 'English',
        'th' => 'ภาษาไทย',
        'ja' => '日本語',        // 3 ← ไม่มีแท็บใน UI แล้ว
        // ... tw, cn, kr, fr, es ...
        // 4 ← ไม่มี 'vn' = เวียดนามพัง
    ];
    return $labels[ $lang ] ?? '';              // 5
}
บรรทัดทำไมต้องมี / ปัญหา
1รับโค้ดภาษาเข้ามา (เช่น 'vn') — helper นี้ถูกเรียกทุกที่ที่ต้องแสดงชื่อภาษา (ปุ่มสลับภาษา, footer)
2ตาราง map โค้ด→ชื่อ · นี่คือแหล่งความจริงที่ต้อง sync กับ qTranslate UI — แต่มันไม่ตรง
3ja ยังอยู่ทั้งที่ UI ไม่มีแท็บ ja แล้ว = โค้ดตาย (dead code) รอยืนยันเพื่อลบ
4ขาด vn — ภาษาที่ 8 ที่ active จริง ตกหล่น = ต้นตอบั๊ก
5?? '' (null coalescing) — ถ้าหา key ไม่เจอคืนค่าว่าง · นี่คือเหตุผลที่บั๊ก "เงียบ" ไม่ error แค่ label หาย ทำให้หายากถ้าไม่รู้ว่าต้องดูตรงไหน

ตัวอย่างประยุกต์

🏢 งานวี: การแก้ lang_label() คืองาน redesign จริงชิ้นแรกที่เข้า GitLab MR ได้ — เล็ก ปลอดภัย ทดสอบง่าย แต่มีคุณค่าจริง (แก้ภาษาเวียดนามที่พังอยู่) · เป็นจุดที่ความเข้าใจ qTranslate แปลงเป็น commit ที่ทีม review แล้ว merge ได้

🌏 อุตสาหกรรม: ปัญหา "แหล่งความจริง 2 ที่ไม่ sync กัน" (config ในโค้ด vs config ในระบบ) เป็นบั๊กคลาสสิกทุกวงการ — เช่น enum ในโค้ดไม่ตรงกับค่าใน DB, รายการประเทศใน dropdown ไม่ตรงกับที่ payment gateway รองรับ · วิธีกันคือ "single source of truth" (แหล่งเดียว) หรือ test ที่จับความไม่ตรงได้

⚠️ ความเสี่ยงระยะยาว (docs/08 §4.1): qTranslate-XT เป็นปลั๊กอิน community fork (แตกจาก qTranslate เดิมที่เลิก maintain) — เสี่ยงหยุดพัฒนา ทีมควรเฝ้าดู + มีแผนสำรอง (ย้าย WPML/Polylang) เผื่ออนาคต · dev ที่ดีรู้ว่า dependency ไหน "เปราะ" และเตรียมทางถอย

🧹 Clean Code ประจำ Week 5 — จัดการ config หลายภาษาให้ปลอดภัย

1. ครอบ helper ปลั๊กอินด้วย function_exists เสมอ

❌ เรียกตรงๆ — ปิดปลั๊กอินแล้วเว็บล่ม

echo lang_label( $lang );
// ถ้า qTranslate ถูกปิด/ถอด
// = Fatal error: undefined function

✅ เช็คก่อนเรียก

if ( function_exists('lang_label') ) {
  echo esc_html( lang_label($lang) );
}
// ปลั๊กอินหาย = แค่ไม่แสดง ไม่ล่ม

2. วน list ภาษาจากแหล่งเดียว ไม่ hardcode ซ้ำ

❌ พิมพ์ภาษาซ้ำหลายที่ — sync ยาก

// ในปุ่มสลับภาษา
echo 'en th tw cn kr fr es';
// อีกไฟล์เขียน list ต่างกัน
// ← ต้นตอบั๊ก lang_label!

✅ list เดียว วนใช้ซ้ำ

$langs = qtranxf_getSortedLanguages();
foreach ( $langs as $l ) {
  echo esc_html( lang_label($l) );
}
// เพิ่ม/ลบภาษาที่เดียว ครบทุกที่
กฎทอง: ทุกครั้งที่ก๊อป list เดียวกันไปวางที่ 2 → รอวันมันไม่ตรงกัน · บั๊ก lang_label นี้เกิดเพราะมี list ภาษา 2 แหล่ง — เอา qTranslate เป็น "แหล่งความจริงเดียว" แล้วดึงมาใช้ ป้องกันปัญหาซ้ำในอนาคต

📝 Quiz — ทดสอบความเข้าใจ

ข้อ 1: ข้อมูลใน DB เก็บเป็น [:en]Sea Candle[:th]ประภาคาร[:] — ทำไมคนเปิด /th/ ถึงเห็นแค่ "ประภาคาร" ไม่เห็นแท็ก?

เพราะ qTranslate เสียบ filter ที่ the_title/the_content/get_field ฯลฯ · filter อ่านภาษาจาก URL (/th/) แล้วตัดเฉพาะส่วน [:th]… ออกมาก่อนแสดง — ข้อมูลเก็บ "รวม" แต่แสดง "แยก" ตามภาษา · ถ้าข้อความไม่ผ่าน filter (เช่น เราประกอบเอง) จะเห็นแท็กดิบทั้งก้อน

ข้อ 2: เอาชื่อสถานที่ไปประกอบใน JSON-LD เอง แล้วแท็ก [:th]…[:en]… โผล่ในผลลัพธ์ — เกิดจากอะไร แก้ยังไง?

เกิดเพราะประกอบข้อความเองนอกเส้นทาง filter อัตโนมัติ (JSON ที่สร้างเองไม่ได้ผ่าน the_title) · แก้ด้วยเรียก qtranxf_use_language($lang, $text) ตัดภาษาเองก่อนใส่ลง JSON · โยง Week 8: JSON-LD ต้องระวังเรื่องนี้เป็นพิเศษเพราะ AI/crawler จะอ่านแท็กดิบไปด้วย

ข้อ 3 (security/production): บั๊ก lang_label() ขาด vn ทำไมถึง "เงียบ" หายาก และหลักการอะไรช่วยกันไม่ให้เกิดซ้ำ?

เงียบเพราะโค้ดใช้ ?? '' — หา key ไม่เจอคืนค่าว่างแทน error → ไม่มีอะไรเตือน แค่ label หายไปเฉยๆ (ถ้าไม่เปิดภาษา vn ดูเองก็ไม่รู้) · หลักกันซ้ำ: single source of truth — ดึง list ภาษาจาก qTranslate ที่เดียว (qtranxf_getSortedLanguages()) แทน hardcode ซ้ำ 2 ที่ · หรือเขียน test ที่ fail เมื่อ 2 แหล่งไม่ตรงกัน

ข้อ 4: ทำไม qTranslate-XT ถูกจัดเป็น "dependency เปราะ" และ dev ควรเตรียมอะไร?
