🧠 แนวคิด & หลักการ

หลักการ 1: "WP_Query = SELECT ของ WordPress"

เปรียบเปรย: ถ้าเคยเขียน SQL — WP_Query คือ SELECT ... FROM wp_posts WHERE ... ORDER BY ... ที่ห่อด้วย PHP array ให้เราไม่ต้องเขียน SQL เอง (และปลอดภัยจาก SQL injection เพราะ WP escape ให้) · เปรียบเปรยอีกแบบ: WP_Query = ใบสั่งของที่บรรณารักษ์ — เราเขียนว่า "ขอหนังสือหมวดท่องเที่ยว 5 เล่มล่าสุด" แล้วบรรณารักษ์ (WP) ไปหยิบมาจากคลัง (DB) ให้

มี 2 ชนิด — ต่างกันตรง "ใครสั่ง":

หลักการ 2: The Loop = เข็มชี้ทีละ post

เปรียบเปรย: The Loop = เข็มเครื่องเล่นแผ่นเสียง · the_post() วางเข็มลงบนโพสต์ปัจจุบัน ทำให้ template tag ทั้งหมด (the_title(), the_field()) รู้ว่า "ตอนนี้กำลังพูดถึงโพสต์ไหน" · วนไปเรื่อยๆ จนหมดแผ่น

โยง Week 1: เราเห็น The Loop เล็กสุดไปแล้ว — บทนี้ต่อยอดเป็น custom query ที่กรอง/เรียง/แบ่งหน้าเองได้

หลักการ 3: เมื่อไหร่ใช้ tax_query / เมื่อไหร่ meta_query

โยงกลับ Week 3 (ทำไม area เป็น taxonomy):

  • tax_query (กรองด้วย taxonomy) — เร็ว เพราะ taxonomy มี index · ใช้เมื่อกรองสิ่งที่เป็น "หมวด/ป้าย" (ประเภท, ย่าน, ฤดู)
  • meta_query (กรองด้วย ACF field) — ช้ากว่า เพราะ scan postmeta · ใช้เมื่อจำเป็นต้องกรองด้วยคุณสมบัติเฉพาะ (วันหมดอายุ, flag) ที่ไม่เหมาะเป็น taxonomy

หลักคิด: ออกแบบ data model ให้สิ่งที่ "ใช้กรองบ่อย" เป็น taxonomy ตั้งแต่แรก (Week 3) → query เร็วโดยไม่ต้องพึ่ง meta_query หนักๆ

ส่วนที่ 1: อ่านโค้ดจริง — df_get_banner_emergency() ทีละส่วน

helper จริงจาก dev-starter (เห็นภาพรวมแล้วใน Week 2) — คราวนี้ผ่าทีละส่วน เพราะมันคือ WP_Query + meta_query ที่ครบเครื่องที่สุดในโปรเจค:

function df_get_banner_emergency() {
  $now = current_time('Y-m-d H:i:s');            // 1
  $q = new WP_Query([
    'post_type'      => 'news',                   // 2
    'posts_per_page' => 1,                        // 3
    'meta_query'     => [
      'relation' => 'AND',                        // 4
      ['key' => 'display_in_banner', 'value' => '1'],        // 5
      ['key' => 'category', 'value' => 'emergency'],         // 6
      ['key' => 'publish_at', 'value' => $now,
       'compare' => '<=', 'type' => 'DATETIME'],            // 7
      ['relation' => 'OR',                                   // 8
        ['key' => 'expire_at', 'value' => $now,
         'compare' => '>=', 'type' => 'DATETIME'],           // 9
        ['key' => 'expire_at', 'compare' => 'NOT EXISTS'],   // 10
      ],
    ],
  ]);
  if ($q->have_posts()) {                         // 11
    $q->the_post();
    $link = get_field('link') ?: get_permalink(); // 12
    $data = ['text' => get_the_title(), 'url' => $link];
    wp_reset_postdata();                          // 13
    return $data;
  }
  return null;                                    // 14
}
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
1current_time() ไม่ใช่ date() — เพราะต้องใช้เวลาโซน Asia/Tokyo ตามตั้งค่า WP ไม่ใช่เวลา server (server อาจอยู่คนละ timezone) · เก็บไว้เทียบกับ publish/expire
2-3query เฉพาะ CPT news · เอาแค่ 1 ข่าว (แบนเนอร์ฉุกเฉินโชว์ทีละอัน)
4'relation' => 'AND' = เงื่อนไขทุกข้อต้องจริงพร้อมกัน (เหมือน AND ใน SQL)
5-6กรอง: ต้องติ๊กให้ขึ้นแบนเนอร์ (display_in_banner=1) และเป็นข่าวฉุกเฉิน (category=emergency) — 2 flag ที่ทีมคอนเทนต์คุมเอง
7publish_at <= now = "ถึงเวลาเผยแพร่แล้ว" · type=DATETIME สำคัญมาก — บอก WP ให้เทียบแบบวันเวลา ไม่ใช่ string (ไม่งั้น "9" > "10" เพราะเทียบตัวอักษร)
8-10ก้อน OR ซ้อนใน AND: "ยังไม่หมดอายุ (expire_at >= now) หรือ ไม่เคยตั้งวันหมดอายุ (NOT EXISTS)" — NOT EXISTS คือกุญแจ: ข่าวที่ไม่กรอก expire_at ต้องไม่ถูกคัดทิ้ง (ไม่มีวันหมด = แสดงตลอด)
11เช็คว่ามีข่าวเข้าเงื่อนไขไหม ก่อนเข้า loop — guard clause (Week 1)
12?: (Elvis operator) — ถ้า field link ว่างให้ใช้ permalink ของข่าวแทน · เขียนสั้นแทน if-else
13🔴 หัวใจ: wp_reset_postdata() — คืนเข็มกลับให้ main query · ลืมบรรทัดนี้ = template tag หลังจากนี้ชี้ post ผิดตัว (ดูส่วนที่ 2)
14ไม่มีข่าวฉุกเฉิน → คืน null → header.php เอาไปเช็คแล้วไม่แสดงแถบ (ไม่ output อะไรเลย ดีกว่าแถบเปล่า)

ส่วนที่ 2: 💥 กับดัก 2 ตัวที่ต้องเจอกับตา

กับดัก 1: ลืม wp_reset_postdata() — เข็มค้างผิดที่

เปรียบเปรย: custom query = ยืมเข็มเครื่องเล่นไปเล่นแผ่นอื่น · ถ้าเล่นเสร็จไม่คืนเข็มกลับที่เดิม แผ่นหลัก (main query) จะเล่นต่อจากตำแหน่งผิด · ผลจริง: the_title() หลัง loop กลายเป็นชื่อข่าวสุดท้ายที่ query ไม่ใช่โพสต์ของหน้านั้น

บั๊กคลาสสิกที่หายากสุดใน WP: หน้าแสดงชื่อผิด แต่โค้ด template tag ดู "ถูก" ทุกอย่าง — เพราะต้นเหตุอยู่ที่ custom loop ก่อนหน้าที่ลืม reset · จำ: ทุก custom WP_Query ที่เข้า loop ต้องจบด้วย wp_reset_postdata() เสมอ

กับดัก 2: orderby meta ทำโพสต์ที่ไม่มี key หลุดหายทั้งชุด

เอกสาร docs/02 §6b มี query แบนเนอร์ที่ตั้งใจเรียงตาม is_pinned (ปักหมุดขึ้นก่อน):

$q = new WP_Query([
  'post_type' => 'news',
  'meta_query' => [ /* ...เงื่อนไขกรอง... */ ],
  'meta_key'  => 'is_pinned',                    // ⚠️ จุดกับดัก
  'orderby'   => ['meta_value_num' => 'DESC', 'date' => 'DESC'],
]);

ปัญหา: การใส่ 'meta_key' => 'is_pinned' เพื่อ orderby ทำให้ WP INNER JOIN ตาราง postmeta ด้วย key นี้ — แปลว่าข่าวที่ไม่เคยเซฟค่า is_pinned (เช่น ข่าวเก่าที่สร้างก่อนเพิ่ม field นี้) จะหลุดจากผลลัพธ์ทั้งหมด → แบนเนอร์ว่างเปล่าทั้งที่มีข่าวฉุกเฉินอยู่จริง!

ทำไมเงียบ: ไม่มี error — แค่ query คืนน้อยกว่าที่ควร · dev ที่ไม่รู้กลไกนี้จะงงว่า "ข่าวหายไปไหน ทั้งที่กรอกแล้ว"

วิธีแก้: ① ตั้ง ACF ให้ is_pinned มี default value (0) เสมอ → ทุกโพสต์มี key = ไม่มีใครหลุด · หรือ ② ใช้ named meta_query clause + 'orderby' => ['clause_name' => 'DESC'] ที่ WP ทำ LEFT JOIN (โพสต์ไม่มี key ไม่หลุด) · แนะนำวิธี ① เพราะเข้าใจง่ายและกันที่ต้นเหตุ

ส่วนที่ 3: listing สำหรับ CPT ที่ has_archive:false

โยง Week 3: Fujisawa มี 14 CPT ที่ has_archive:false = ไม่มีหน้ารวมอัตโนมัติ · ทำหน้า listing เองด้วย custom WP_Query + pagination:

$paged = get_query_var('paged') ?: 1;            // 1
$q = new WP_Query([
  'post_type'      => 'ssws__shop',              // 2 ที่พัก
  'posts_per_page' => 10,
  'paged'          => $paged,                     // 3
  'meta_query'     => [['key'=>'ssws__shop__flag','value'=>'1']], // 4 กรอง flag!
]);
while ($q->have_posts()): $q->the_post();
  get_template_part('template-parts/card-hotel'); // 5
endwhile;
wp_pagenavi(['query' => $q]);                     // 6
wp_reset_postdata();                              // 7
บรรทัดทำไมต้องมี
1,3อ่านเลขหน้าปัจจุบันจาก URL แล้วส่งให้ query — ขาด paged = ทุกหน้าแสดงชุดเดิม (pagination พัง)
4กรอง flag เสมอ (Week 3!) — ไม่งั้นที่พักที่ทีมปิดไว้โผล่บนเว็บ
5get_template_part() = แยกการ์ดออกเป็นไฟล์ reuse ได้ — ไม่ก๊อป markup ซ้ำ (Clean Code)
6wp_pagenavi() = ปลั๊กอินทำเลขหน้า 1 2 3 · ต้องส่ง $q เพราะเป็น custom query ไม่ใช่ main
7reset เข็ม (กับดัก 1!)

🧹 Clean Code ประจำ Week 6 — เขียน query/loop ให้สะอาด

1. แยก query ออกจาก markup (logic ไม่ปนกับหน้าตา)

❌ query ปนกลาง HTML — อ่านยาก reuse ไม่ได้

<div class="list">
<?php $q = new WP_Query([...50 บรรทัด...]);
while(...): the_post(); ?>
  <div>...markup...</div>
<?php endwhile; ?>
</div>

✅ query ข้างบน · markup ใช้ template part

<?php $hotels = ssws_get_hotels($paged); ?>
<div class="list">
<?php while($hotels->have_posts()):
  $hotels->the_post();
  get_template_part('parts/card-hotel');
endwhile; wp_reset_postdata(); ?>
</div>

2. wp_reset_postdata() ติดคู่ custom loop เสมอ (มือจำ)

❌ ลืม reset — บั๊กเข็มค้าง

while($q->have_posts()): $q->the_post();
  the_title();
endwhile;
// จบเลย → the_title() ถัดไปเพี้ยน

✅ เขียน reset พร้อม loop ทันที

while($q->have_posts()): $q->the_post();
  the_title();
endwhile;
wp_reset_postdata();   // พิมพ์คู่กันเสมอ
กฎทอง: ทุกครั้งที่พิมพ์ new WP_Query ให้พิมพ์ wp_reset_postdata() ต่อท้ายทันทีก่อนเขียน loop ตรงกลาง — ให้มันเป็นคู่ที่แยกกันไม่ได้ในนิสัย เหมือน { } ที่เปิดต้องปิด

📝 Quiz — ทดสอบความเข้าใจ

ข้อ 1: the_title() หลัง custom loop แสดงชื่อผิด (เป็นชื่อข่าวสุดท้ายที่ query) — เกิดจากอะไร แก้ยังไง?

ลืม wp_reset_postdata() หลัง custom loop — เข็ม (global $post) ค้างอยู่ที่โพสต์สุดท้ายของ custom query ไม่ถูกคืนให้ main query · แก้: เรียก wp_reset_postdata() ทันทีหลัง endwhile ของทุก custom WP_Query ที่เข้า loop

ข้อ 2: ทำไม tax_query เร็วกว่า meta_query? แล้วควรออกแบบ data model ยังไงให้ query เร็ว?

taxonomy มี index ในตาราง terms/term_relationships → lookup เร็ว · meta_query ต้อง scan postmeta ที่ไม่ได้ index มาเพื่อการนี้ → ช้ากว่าเมื่อข้อมูลโต · ออกแบบให้ "สิ่งที่ใช้กรองบ่อย" (ประเภท ย่าน ฤดู) เป็น taxonomy ตั้งแต่ Week 3 → query เร็วโดยไม่ต้องพึ่ง meta_query หนัก

ข้อ 3 (security/production): query แบนเนอร์ตั้ง orderby ตาม is_pinned แล้วข่าวฉุกเฉินบางอันหายไป — อธิบายกลไก และวิธีแก้

การใส่ meta_key เพื่อ orderby ทำให้ WP INNER JOIN postmeta ด้วย key นั้น → โพสต์ที่ไม่เคยเซฟ is_pinned (ข่าวเก่า/สร้างก่อนมี field) ถูกตัดออกทั้งหมด → แบนเนอร์อาจว่างทั้งที่มีข่าวด่วน = ปัญหา production ร้ายแรง (ข้อมูลฉุกเฉินไม่ถึงผู้ใช้) · แก้: ตั้ง ACF default value ให้ is_pinned มีค่าเสมอ หรือใช้ named meta_query clause ที่ LEFT JOIN · บทเรียน: orderby บน optional field ต้องระวัง JOIN ที่ซ่อนอยู่

ข้อ 4: ในโค้ด df_get_banner_emergency ทำไมต้องมีเงื่อนไข expire_at NOT EXISTS ใน OR ด้วย?

เพราะข่าวที่ไม่กรอกวันหมดอายุ (ตั้งใจให้แสดงไม่มีกำหนด) จะไม่มี key expire_at เลย · ถ้ามีแค่เงื่อนไข expire_at >= now ข่าวพวกนี้จะไม่เข้าเงื่อนไข = หลุดหาย · NOT EXISTS ครอบคลุมเคส "ไม่มีวันหมด = แสดงตลอด" · หลักคิด: เขียน query ต้องคิดถึง "โพสต์ที่ field ว่าง" เสมอ (โยงกับดัก orderby ข้อ 3 — ธีมเดียวกัน)
