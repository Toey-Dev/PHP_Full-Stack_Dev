🔑 คีย์เวิร์ดภาพจำ

📛 CPT = คำนาม (ชนิดของสิ่ง มีหน้าของตัวเอง) 🏷 Taxonomy = ป้ายจัดหมวด (แชร์ข้าม CPT ได้) ⚙️ Field = คุณสมบัติเฉพาะตัว 🚦 flag = สวิตช์ Show/Hide (listing ต้องกรองเสมอ) 👯 whattosee vs sstr_spot = ฝาแฝดที่ต้องถามทีม 🔤 prefix = นามสกุลบอกตระกูล (sslc/ssws/ssnb)

🧠 แนวคิด & หลักการ

หลักการ 1: "ออกแบบข้อมูลก่อน ออกแบบหน้า"

ปัญหาที่มันแก้: มือใหม่เริ่มจาก "หน้าตาอยากให้เป็นแบบนี้" แล้วยัดข้อมูลตามหลัง → พอเว็บโต ข้อมูลซ้ำ กรองไม่ได้ query ช้า · มือโปรเริ่มจาก "ข้อมูลนี้มีโครงสร้างยังไง สัมพันธ์กับอะไร" แล้วค่อยสร้างหน้ามาแสดง — ลำดับที่ถูกคือ data model → URL → template → style (สวนทางมือใหม่ทุกข้อ)

เปรียบเปรย: data model = พิมพ์เขียวบ้าน · จะวางท่อน้ำ/สายไฟที่ไหนต้องรู้ตั้งแต่ก่อนเทปูน — จะมาทุบผนังเดินท่อใหม่ตอนบ้านเสร็จแล้วคือหายนะ · การ redesign Fujisawa ก็เหมือนกัน: เข้าใจพิมพ์เขียวข้อมูลเดิมก่อน ค่อยรีโนเวทหน้าตา

หลักการ 2: 3 คำถามที่ตัดสินว่าสิ่งหนึ่งเป็นอะไร

ทุกครั้งที่เจอ "ข้อมูลชิ้นใหม่" ถาม 3 คำถามนี้ตามลำดับ:

คำถามถ้าใช่ = ตัวอย่างใน Fujisawa
มันเป็น "สิ่ง" ที่มีหน้ารายละเอียดของตัวเองไหม?CPTสถานที่, ร้านอาหาร, โรงแรม, ข่าว, อีเวนต์
มันเป็น "ป้าย" ที่ใช้จัดกลุ่ม/กรอง และใช้ซ้ำได้หลายสิ่งไหม?Taxonomyประเภทสถานที่, ย่าน, ฤดูกาล
มันเป็น "คุณสมบัติเฉพาะตัว" ของสิ่งนั้นไหม?Field (ACF)เวลาเปิด-ปิด, พิกัด, ราคา, ชื่อภาษาอังกฤษ

หลักการ 3: เมื่อไหร่เป็น Taxonomy / เมื่อไหร่เป็น Field (จุดที่พลาดบ่อยสุด)

"ย่าน (area)" ควรเป็น taxonomy หรือ ACF select field? — คำถามคลาสสิกที่แยกมือใหม่กับมือโปร คำตอบ: taxonomy เพราะ:

  • ใช้ กรอง ("โรงแรมทั้งหมดในย่าน Enoshima") → taxonomy มี index ให้ query เร็ว · ถ้าเป็น field ต้อง meta_query LIKE = ช้า
  • ได้ หน้า archive ฟรี — URL /area/enoshima/ ทำงานเองโดยไม่ต้องเขียน
  • ใช้ ซ้ำข้าม CPT ได้ — ย่านเดียวกันผูกได้ทั้งโรงแรม ร้านอาหาร สถานที่

Counter-example: ถ้าทำ area เป็น ACF select field — query "ทุกอย่างในย่านนี้" จะช้าลงเรื่อยๆ เมื่อข้อมูลโต, ไม่มีหน้า archive, และต้องก๊อป choices ไปทุก CPT (แก้ที่เดียวไม่พอ) · นี่คือเหตุผลที่ Fujisawa แยก ssnb__mst_area, ssws__mst_area เป็น taxonomy

🚦 ระบบ flag — สวิตช์ Show/Hide ที่ต้องกรองเสมอ

เปรียบเปรย: ทุก CPT ของ Fujisawa มี field ชื่อลงท้าย __flag (เช่น sslc__shop__flag) — มันคือสวิตช์ไฟของแต่ละโพสต์ ทีมคอนเทนต์ปิดสวิตช์เพื่อ "ซ่อนชั่วคราวโดยไม่ต้องลบ" (ร้านปิดปรับปรุง, อีเวนต์จบแล้ว)

💥 กับดักที่ต้องระวังทุก listing: เวลา query รายการมาแสดง ต้องกรอง flag == true เสมอ ไม่งั้นโพสต์ที่ทีมตั้งใจซ่อนจะโผล่บนเว็บ · counter-example จริง: ลืมกรอง flag → ร้านที่ปิดกิจการไปแล้วยังโชว์อยู่หน้าเว็บ ลูกค้าเดินทางไปเก้อ = เรื่องร้องเรียน (Week 6 จะเขียน meta_query กรอง flag จริง)

ส่วนที่ 3: ปริศนาของจริง 3 ข้อ (ที่ dev ต้องเคลียร์ก่อนเริ่ม)

👯 ปริศนา 1: whattosee vs sstr_spot — สถานที่ 2 CPT ซ้อนกัน

Fujisawa มี CPT ที่เป็น "สถานที่" 2 ตัว:

whattoseesstr_spot
LabelWhat to seeTourism spots
has_archivetruefalse
taxonomyspot_catผูกกับ sstr_grpd_route (เส้นทาง)
ดีไซน์ที่ map มาExploreSpots / SpotDetailจุดในเส้นทาง (Itineraries)

ดีไซน์ SpotDetail ของเรา map มาที่ whattosee — แต่ sstr_spot ก็มีข้อมูลจริง 10+ รายการ นี่คือ Risk R7 ในเอกสาร: ถ้า query ผิด CPT = หน้าไม่มีข้อมูล · ต้องถามทีมก่อนว่าบทบาทต่างกันยังไง จะ map ไปตัวไหน

🪦 ปริศนา 2: ss_whattosee — ศพ template ที่ค้างในธีม

ในธีมจริงมีไฟล์ single-ss_whattosee.php และ templates/ss_whattosee.php — แต่ CPT ชื่อ ss_whattosee ไม่มีอยู่แล้ว · สาเหตุ: เดิม CPT ชื่อนี้ ถูกrename เป็น whattosee ผ่านฟีเจอร์ "Migrate posts" ของ CPT UI (โพสต์ย้ายไป type ใหม่หมด) แต่ไฟล์ template เก่าค้างไว้ → ลบได้ปลอดภัย (ยืนยันจาก CPT UI แล้ว) · บทเรียน: ⚠️ ห้ามแก้ slug ของ CPT เล่นๆ — การเปลี่ยน slug = register type ใหม่ เนื้อหาเดิมหลุด (ต้องติ๊ก migrate เสมอ = Risk R1)

📦 ปริศนา 3: has_archive:false 14 ตัว + can_export:false ทุกตัว

has_archive:false (14/18 CPT): WordPress ไม่สร้างหน้ารวมให้อัตโนมัติ → ต้องทำ listing เองผ่าน page template + WP_Query หรือปลั๊กอิน Custom Post Type Archives (Week 6 ทำจริง)

can_export:false (ทุก CPT): การ export Tools → All content (WXR) จะไม่มีเนื้อหา CPT ติดมาเลย (ได้แค่ page/post/attachment) → เวลา migrate ข้อมูลไป sandbox ต้องพึ่ง DB dump หรือ REST API ไม่ใช่ WXR ปกติ (Risk R2 — มีผลกับ Mission ที่ต้องมีข้อมูลทดสอบ)

🧹 Clean Code ประจำ Week 3 — ออกแบบ data model ให้อ่านออก

1. ตั้งชื่อ slug สื่อความหมาย + มีระบบ

❌ ชื่อมั่ว เดาตระกูลไม่ได้

register_post_type('type1', ...);
register_post_type('shop', ...);
register_post_type('shop2', ...);
// shop กับ shop2 ต่างกันยังไง?

✅ prefix บอกตระกูล + ชื่อบอกหน้าที่

register_post_type('sslc__shop',...); // อาหาร
register_post_type('ssnb__shop',...); // บาร์
register_post_type('ssws__shop',...); // ที่พัก
// เห็นชื่อ = รู้ตระกูลทันที

2. เลือก taxonomy แทน field เมื่อข้อมูลใช้กรอง

❌ area เป็น ACF select — query ช้า

// เก็บย่านเป็น field
'meta_query' => [[
  'key'=>'area','value'=>'enoshima'
]]  // LIKE บน postmeta = ช้า
// + ไม่มีหน้า /area/enoshima/

✅ area เป็น taxonomy — index เร็ว + archive ฟรี

'tax_query' => [[
  'taxonomy'=>'ssws__mst_area',
  'terms'=>'enoshima'
]]  // ใช้ index = เร็ว
// + ได้ /area/enoshima/ อัตโนมัติ
กฎทอง: การตัดสิน CPT/Taxonomy/Field คือการตัดสินใจที่ย้อนกลับยากที่สุดในโปรเจค — เพราะข้อมูลถูกกรอกเข้าไปแล้วเป็นพันชิ้น การย้ายทีหลังคือ migrate + เสี่ยงข้อมูลหาย · คิดให้จบตั้งแต่พิมพ์เขียว

📝 Quiz — ทดสอบความเข้าใจ

ข้อ 1: "ราคาห้องพัก" ควรเป็น CPT, Taxonomy หรือ Field? เพราะอะไร

Field — เพราะมันเป็นคุณสมบัติเฉพาะตัวของโรงแรมแต่ละแห่ง ไม่ได้ใช้ "จัดกลุ่ม" (ไม่มีใครกรอง "โรงแรมราคา 3,200 บาทพอดี") และไม่มีหน้ารายละเอียดของ "ราคา" เอง · ถ้าอยากกรองเป็นช่วงราคา (ถูก/กลาง/แพง) ค่อยเพิ่ม taxonomy "ระดับราคา" แยกต่างหาก

ข้อ 2: ทำไม "ย่าน (area)" เป็น taxonomy ดีกว่า ACF select field?

เพราะ area ใช้ "กรอง" (โรงแรมทั้งหมดในย่านนี้) + ใช้ซ้ำข้าม CPT — taxonomy ให้ 3 อย่างที่ field ให้ไม่ได้: ① index ทำให้ query เร็ว (field ต้อง meta_query LIKE = ช้า) ② หน้า archive /area/enoshima/ ฟรี ③ แก้ term ที่เดียวมีผลทุก CPT ที่ผูก

ข้อ 3 (security/production): can_export:false ทุก CPT ทำให้ export WXR ว่าง — กระทบการเตรียม sandbox ยังไง และวิธีแก้?

WXR (Tools → Export) จะไม่มีเนื้อหา CPT ติดมา → ถ้าพึ่ง WXR อย่างเดียว sandbox จะไม่มีข้อมูลทดสอบ template · วิธีแก้: ① DB dump ผ่าน Fujisawa migration system (ได้ครบสุด) ② REST API /wp-json/wp/v2/{cpt} — แต่ต้องเปิด show_in_rest ที่ field group ก่อน ไม่งั้น acf:[] ว่าง (Risk R3) ③ WP All Export (CSV) · บทเรียน production: อย่าสมมุติว่า "export = ได้ทุกอย่าง" ต้องรู้ข้อจำกัดของเครื่องมือ

ข้อ 4: ทำไมการเปลี่ยน slug ของ CPT ใน CPT UI เล่นๆ ถึงอันตราย (เคส ss_whattosee)?

เพราะ slug คือค่าใน post_type ที่ผูกโพสต์ทุกชิ้นเข้ากับ type · เปลี่ยน slug = register type ใหม่ที่ไม่มีโพสต์เดิมผูกอยู่ → เนื้อหาเดิมทั้งหมด "หลุด" (ยังอยู่ใน DB แต่ชี้ไป type ที่ไม่มีแล้ว) หน้า 404 ทั้งหมวด · ทางถูก: ต้องติ๊ก "Migrate posts to renamed post type" เสมอ + สำรอง DB ก่อน (Risk R1)
