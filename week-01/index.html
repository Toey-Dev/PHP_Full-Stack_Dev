🧠 แนวคิด & หลักการ

หลักการ 1: เว็บ dynamic = "แยกข้อมูลออกจากหน้าตา"

ปัญหาที่มันเกิดมาแก้: ลองนึกภาพดีไซน์ HTML 44 หน้าของ Fujisawa ถ้าใช้แบบ static ตรงๆ — ทีมคอนเทนต์อยากเพิ่มร้านอาหาร 1 ร้าน ต้องมาตามหา dev ให้แก้ไฟล์ HTML แล้วอัปโหลดใหม่ ทุกครั้ง ทุกภาษา (×8!) เว็บแบบนี้เรียกว่า "สวยแต่ตาย" — เนื้อหาถูกฝังตายอยู่ในโค้ด

เปรียบเปรย: เว็บ static = ป้ายเขียนมือ อยากเปลี่ยนข้อความต้องทาสีเขียนใหม่ทั้งป้าย · เว็บ dynamic (WordPress) = ป้ายไฟ LED ตัวป้าย (template) ติดตั้งครั้งเดียว ข้อความวิ่งมาจากแผงควบคุม (wp-admin) — คนเปลี่ยนข้อความไม่ต้องรู้ว่าหลอดไฟต่อกันยังไง

หลักคิดเบื้องหลังคือ Separation of Concerns แยก 3 อย่างออกจากกันเด็ดขาด:

ส่วนคืออะไรใครดูแล
🖼 หน้าตา (Template)โครง HTML + PHP ที่มี "ช่องว่าง" รอเสียบข้อมูลdev (เรา) — เขียนครั้งเดียว
🗄 ข้อมูล (Database)เนื้อหาจริง: ชื่อร้าน รูป เวลาเปิด-ปิด 8 ภาษาฐานข้อมูล — แหล่งความจริงเดียว
🚪 ประตูแก้ข้อมูล (wp-admin)ฟอร์มหลังบ้านสำหรับเพิ่ม/แก้/ลบเนื้อหาทีมคอนเทนต์ — ไม่ต้องเขียนโค้ด

ประโยคเดียวที่สรุปงาน redesign ทั้งโปรเจค (มาจากเอกสารส่งมอบจริง): "ดีไซน์ = โครง markup · ฐานข้อมูล = แหล่งความจริง · เนื้อหาแก้ผ่าน wp-admin"

หลักการ 2: จะเข้าใจ WordPress ต้องไล่ "เส้นทางข้อมูล" ให้ออก

คนที่ "ใช้ WP เป็น" จำว่าต้องคลิกอะไร — พอเจอโจทย์ใหม่จะไปต่อไม่ได้ คนที่ "เข้าใจ WP" รู้ว่าข้อมูลไหลจากไหนไปไหน — เจอโจทย์ใหม่ก็ไล่เส้นทางแล้วหาจุดเสียบโค้ดเองได้ ทั้งระบบ WordPress ตั้งอยู่บน 3 เสา ตามเส้นทางข้อมูลพอดี:

  1. Data: เนื้อหาทุกชนิดเก็บในตาราง wp_posts (+ ข้อมูลเสริมใน wp_postmeta)
  2. Logic: การต่อเติมทุกอย่างทำผ่าน hook — เสียบโค้ดเพิ่ม ไม่แก้ของเดิม
  3. Presentation: WP เลือกไฟล์ template ให้อัตโนมัติตามกติกาชื่อไฟล์ (template hierarchy)

หลักการ 3: เมื่อไหร่ใช้ / เมื่อไหร่ไม่ใช้

ถ้าเนื้อหาแทบไม่เคยเปลี่ยนและไม่มีทีมคอนเทนต์ — static HTML ก็พอ (เร็วกว่า ปลอดภัยกว่า ดูแลถูกกว่า) WordPress คุ้มเมื่อคนที่ไม่เขียนโค้ดต้องแก้เนื้อหาเองบ่อยๆ — ซึ่งคือเคส Fujisawa เป๊ะ: ข่าวออกทุกสัปดาห์ อีเวนต์เปลี่ยนตามฤดู ร้านเปิด-ปิด และทีมคอนเทนต์ดูแล 8 ภาษา

ส่วนที่ 1: เสา Data — "ทุกอย่างคือ wp_posts"

เปรียบเปรย: WordPress ไม่ได้สร้างตู้เอกสารใหม่ทุกครั้งที่มีเนื้อหาชนิดใหม่ — มันใช้ตู้เอกสารตู้เดียวชื่อ wp_posts แล้วติดป้ายแยกลิ้นชักด้วยคอลัมน์เดียวคือ post_type (ลิ้นชัก page, ลิ้นชัก news, ลิ้นชักรูปภาพ...) ส่วนข้อมูลเสริมของแต่ละแฟ้ม (เวลาเปิด-ปิด, พิกัด, ราคา) เขียนใส่โพสต์อิทแปะข้างแฟ้ม = ตาราง wp_postmeta (เก็บเป็นคู่ key–value ผูกกับ post_id)

กรอกอะไรใน wp-admin แล้วมันไปไหน?

นี่คือคำถามที่เชื่อม "หน้าจอที่วีใช้ทุกวัน" เข้ากับ "ฐานข้อมูลจริง" — ตารางนี้คือแผนที่:

สิ่งที่กรอกใน wp-adminไปเก็บที่
ช่องหัวข้อ (Title)wp_posts.post_title
กล่องเนื้อหาหลัก (Editor)wp_posts.post_content
Featured imageรูปเป็นแถวใหม่ (post_type=attachment) + โพสต์อิท _thumbnail_id ชี้หากัน
ทุกช่องของ ACF (เวลาเปิด-ปิด, พิกัด, FAQ...)wp_postmeta — 1 field = 1 แถว key–value
หมวด/แท็ก (taxonomy)ตระกูล wp_terms (Week 3 เจาะเต็ม)
ทำไมต้องรู้แผนที่นี้: เวลาหน้าเว็บแสดงข้อมูลผิด จะได้รู้ว่าไปไล่ดูที่ไหน — "ชื่อผิด" ดูที่ wp_posts / "เวลาเปิด-ปิดผิด" ดูที่ postmeta / "อยู่ผิดหมวด" ดูที่ terms — debug เร็วขึ้น 10 เท่า

🌏 ตัวอย่างจริง #2 — อุตสาหกรรม

WooCommerce (ระบบ e-commerce ที่ใช้มากที่สุดใน WP) ใช้หลักเดียวกันเป๊ะ: สินค้า = post_type product, ออเดอร์ = post_type shop_order, ราคา = โพสต์อิท _price ใน postmeta — เข้าใจหลักนี้ที่เดียว อ่านออกทั้ง ecosystem

🔍 Why — ทำไมออกแบบเป็นตู้เดียว (และถ้าไม่ทำแบบนี้จะเป็นยังไง)

เหตุผล: WP เป็น CMS ที่ไม่รู้ล่วงหน้าว่าผู้ใช้จะมีเนื้อหาชนิดไหน — ถ้าเนื้อหาชนิดใหม่ต้อง CREATE TABLE ใหม่ทุกครั้ง ปุ่ม "Add Post Type" ใน CPT UI จะไม่มีทางทำงานได้ทันที การใช้ตู้เดียว+ป้ายลิ้นชักทำให้เพิ่มชนิดเนื้อหาได้ในพริบตา แลกกับ query ที่ซับซ้อนขึ้นเมื่อกรองด้วย meta เยอะๆ (trade-off นี้จะเจอเต็มๆ ใน Week 6)

Counter-example จากทีมวีเอง: ใน DB เดียวกันมีตาราง custom wp_fjs_contact และ wp_import_* 18 ตาราง — ทีม fujisawa เลือก "สร้างตู้ใหม่" เพราะข้อมูลพวกนี้ (ฟอร์มติดต่อ, ข้อมูล migrate) ไม่ใช่เนื้อหาที่ทีมคอนเทนต์แก้ผ่าน admin → เส้นแบ่งที่ใช้ตัดสิน: คนทำคอนเทนต์ต้องแก้ผ่าน admin ไหม? ใช่ = wp_posts / ไม่ใช่ (ข้อมูลระบบ) = ตารางเอง

ส่วนที่ 2: เสา Logic — Hook = ปลั๊กไฟ

เปรียบเปรย: ทุกครั้งที่ WP โหลดหน้า มันวิ่งไปตามไทม์ไลน์เดิมเสมอ (โหลดปลั๊กอิน → init → query → render → ปิดท้าย) และตลอดทางมันเปิดรูปลั๊กไฟไว้เป็นระยะ (ผ่านคำสั่ง do_action('ชื่อจุด')) ใครอยากให้อะไรทำงาน ณ จุดไหน ก็เอาโค้ดไปเสียบปลั๊กที่จุดนั้น (add_action()) — โดยไม่ต้องผ่าสายไฟหลัก (ไม่แก้ไฟล์ WP core / parent theme เลย)

หลักคิดเบื้องหลังคือ Open/Closed Principle: ระบบที่ดี "ปิดรับการแก้ไข แต่เปิดรับการต่อขยาย" เพราะ WP core และ theme หลักถูกอัปเดตทับตลอดเวลา — แก้มันตรงๆ = งานหายตอนอัปเดต

Hook มี 2 ชนิด ต่างกันตรง "ของที่ไหลผ่านปลั๊ก":

actionfilter
ทำหน้าที่"ทำอะไรบางอย่าง ณ จุดนี้" (พ่น HTML, register ของ)"รับค่ามาแปลง แล้วส่งคืน" (แก้ title, ตัดคำ)
ต้องคืนค่าไหมไม่ต้องต้องเสมอ — WP เอาค่าที่คืนไปใช้ต่อจริงๆ
ถ้าพลาดของไม่โผล่ / โผล่ผิดที่ลืม return = คืน null = ข้อมูลส่วนนั้นหายทั้งเว็บ

🌏 ตัวอย่างจริง #1 — action ในโค้ดที่รันอยู่จริง (welovefukushima)

ไฟล์ inc/schema.php เสียบ Schema.org เข้า <head> เฉพาะหน้า sightseeing เพื่องาน GEO/AEO โดยไม่แตะ theme หลัก (Blocksy) แม้แต่บรรทัดเดียว:

function wlf_sightseeing_schema() {              // 1
    if ( ! is_singular( 'sightseeing' ) ) {      // 2
        return;                                  // 3
    }
    // ... สร้าง JSON-LD TouristAttraction ...   // 4
}
add_action( 'wp_head', 'wlf_sightseeing_schema' ); // 5
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
1ประกาศฟังก์ชันชื่อขึ้นต้น wlf_ — WP โหลดทุกปลั๊กอิน/theme ใน global เดียวกัน ชื่อซ้ำ = เว็บล่มทันที (fatal error) prefix คือประกันชนกัน
2ด่านตรวจ: hook wp_head ยิงทุกหน้าของเว็บ — ถ้าไม่เช็คก่อน schema "สถานที่ท่องเที่ยว" จะโผล่ในหน้า contact ด้วย = ข้อมูลผิดความหมาย Google/AI สับสน
3return เปล่า = "หน้านี้ไม่ใช่ธุระของฉัน จบเลย" — เทคนิค guard clause (ดูใน Clean Code ท้ายบท)
4งานจริงของฟังก์ชัน — ทำหลังผ่านด่านแล้วเท่านั้น
5เสียบปลั๊ก: "พอถึงจุด wp_head ให้เรียกฟังก์ชันนี้" — เลือก wp_head เพราะ schema ต้องอยู่ใน <head> และกติกา WP บังคับให้ทุก theme เปิดรูปลั๊กนี้ไว้ตรงนั้น

🌏 ตัวอย่างจริง #2 — filter: เปลี่ยนของทั้งเว็บจากจุดเดียว

add_filter( 'excerpt_more', 'wlf_excerpt_more' );  // 1
function wlf_excerpt_more( $more ) {               // 2
    return ' …อ่านต่อ';                             // 3
}
บรรทัดทำไมต้องมี
1เสียบ filter ชื่อ excerpt_more = จุดที่ WP ถามว่า "ตัวย่อเนื้อหาจะลงท้ายด้วยอะไร" — แก้ที่จุดเดียว มีผลทุกหน้าทั้งเว็บ ไม่ต้องไล่แก้ template ทีละไฟล์
2รับค่าเดิม ($more = "[...]" ของ WP) เข้ามา — filter คือ "สายพานแปลงค่า" ต้องมีทางเข้า
3ต้องคืนค่าเสมอ — WP เอาสิ่งที่ return ไปแสดงจริง
💥 บั๊กคลาสสิกอันดับ 1 ของ filter: ลืม return → ฟังก์ชันคืน null → ตัวย่อเนื้อหาทั้งเว็บกลายเป็นค่าว่างทันที — Mission ท้ายบทจะให้ทำพังดูหนึ่งครั้งด้วยตาตัวเอง จะได้จำไปตลอดชีวิต

🔍 Why — ทำไมห้ามแก้ core / parent theme ตรงๆ

Counter-example: สมมุติแก้ header.php ของ theme หลักตรงๆ เพื่อเพิ่มแบนเนอร์ → วันที่ theme ออกอัปเดต ไฟล์ถูกทับ งานหายทั้งหมดแบบไม่มี warning — นี่คือเหตุผลที่ welovefukushima ต้องเป็น child theme และทุก logic เสียบผ่าน hook และเหตุผลเดียวกันจะตอบ Week 2 ว่าทำไม CPT ควรอยู่ใน plugin: โค้ดต้องอยู่รอดเมื่อสิ่งที่มันเกาะถูกอัปเดตหรือเปลี่ยน

โยงกลับเป้า dynamic: hook คือกลไกที่ทำให้เรา "เพิ่มความสามารถ" ให้เว็บได้เรื่อยๆ โดยของเดิมไม่พัง — งาน redesign Fujisawa ทั้งหมดจะยืนอยู่บนหลักนี้

ส่วนที่ 3: เสา Presentation — Template Hierarchy: บันไดเลือกไฟล์

เปรียบเปรย: WP ไม่มีไฟล์ config ที่บอกว่า URL ไหนใช้ template ไหน — มันใช้วิธีไล่บันไดหาไฟล์ตามลำดับชื่อ: ดูก่อนว่า request คืออะไร (หน้า single ของ CPT ไหน? หน้า archive? หน้า page?) แล้วเดินลงบันไดจากชื่อไฟล์ที่เจาะจงที่สุด → กว้างที่สุด เจอไฟล์แรกที่มีจริง = ใช้เลย หยุดหา ถ้าไม่เจอสักไฟล์ จบที่ index.php เสมอ (หลักคิด: convention over configuration)

นี่คือเหตุผลที่ "แค่ตั้งชื่อไฟล์ถูก ก็ได้หน้าใหม่ฟรี" — และเหตุผลที่ตั้งชื่อผิดตัวเดียวแล้วนั่งงงว่าทำไมหน้าไม่เปลี่ยน

เครื่องมือ debug: ปลั๊กอิน Query Monitor (ติดตั้งใน Mission) บอกตรงๆ ว่าหน้านี้ใช้ template ไฟล์ไหน — เลิกเดาได้เลย และหน้าจอ CPT UI → Registered Types ก็มีคอลัมน์ Template Hierarchy บอกบันไดของทุก CPT

ส่วนที่ 4: ต่อจิ๊กซอว์ — ดึงข้อมูลจาก DB มาแสดงจริง (ชิมลาง Week 6)

สามเสามารวมกันตรงนี้: template (เสา 3) ดึงข้อมูล (เสา 1) มาแสดง — นี่คือโค้ดที่เล็กที่สุดที่ "dynamic ครบวงจร":

<?php while ( have_posts() ) : the_post(); ?>   // 1
  <h2><?php the_title(); ?></h2>                // 2
  <div><?php the_content(); ?></div>            // 3
<?php endwhile; ?>                              // 4
บรรทัดทำไมต้องมี
1The Loop — "ตราบใดที่ยังมีแฟ้มในมือ หยิบแฟ้มถัดไปขึ้นมาเปิด" (the_post() = ชี้เข็มไปที่แฟ้มปัจจุบัน) — WP เตรียมแฟ้มมาให้แล้วจาก URL ที่ผู้ใช้ขอ (main query)
2the_title() = "พิมพ์ชื่อเรื่องของแฟ้มที่เข็มชี้อยู่" — ค่ามาจาก wp_posts.post_title ที่ทีมคอนเทนต์กรอกใน admin
3the_content() = เนื้อหาจากกล่อง editor (wp_posts.post_content)
4ปิด loop — HTML ระหว่างบรรทัด 1-4 จะถูกพิมพ์ซ้ำต่อ 1 แฟ้ม (หน้า archive มีหลายแฟ้ม = พิมพ์หลายรอบ)

พิสูจน์ความ dynamic ด้วยตาตัวเอง (ทำใน Mission): แก้ title ของโพสต์ใน wp-admin → กด refresh หน้าเว็บ → ข้อความเปลี่ยนเอง โดยไม่ได้แตะโค้ดสักบรรทัด — เพราะ template เป็นแค่ "ช่องว่าง" ข้อมูลจริงอยู่ใน DB เสมอ

อย่าเพิ่งกังวล syntax ของ loop — Week 6 เจาะเต็มทั้ง WP_Query / tax_query / pagination ตอนนี้ขอแค่เห็นภาพว่า "template ดึงข้อมูลจากตู้มาเสียบช่องว่าง" ก็พอ

🧹 Clean Code ประจำ Week 1 — เขียน hook แบบมืออาชีพ

3 รูปแบบที่ใช้ได้ทันทีใน Mission สัปดาห์นี้:

1. ตั้งชื่อฟังก์ชันมี prefix เสมอ (กันเว็บล่ม)

❌ ชื่อกลางๆ — รอวันชนกับปลั๊กอินอื่น

function add_meta_tag() { ... }
// ปลั๊กอินอื่นประกาศชื่อเดียวกัน
// = Fatal error เว็บล่มทั้งเว็บ

✅ prefix โปรเจค — ไม่มีทางชน

function whl_add_meta_tag() { ... }
// whl = wp-hooks-lab
// (theme fujisawa จริงใช้ fjsw_)

2. Guard clause — เช็คเงื่อนไขแล้วออกก่อน ไม่ซ้อน if

❌ ซ้อน if — โค้ดจริงอยู่ลึก อ่านยาก

function whl_schema() {
  if ( is_singular('sightseeing') ) {
    if ( ! is_admin() ) {
      // งานจริงอยู่ลึก 2 ชั้น
    }
  }
}

✅ ไม่ใช่ธุระ = ออกทันที

function whl_schema() {
  if ( ! is_singular('sightseeing') ) {
    return;   // ไม่ใช่หน้านี้ → ออกเลย
  }
  if ( is_admin() ) {
    return;   // อยู่หลังบ้าน → ออกเลย
  }
  // งานจริงอยู่ชั้นนอกสุด อ่านง่าย
}
🔄 กฎการ "พลิกเงื่อนไข" (จุดที่คนสับสนที่สุด): เวลาเปลี่ยนจากซ้อน if → guard clause ต้องพลิกทุกเงื่อนไขให้กลายเป็น "เงื่อนไขที่จะไล่ออก" — ตัวไหนเคยมี ! จะหายไป ตัวไหนไม่เคยมีจะได้ ! มา
if ( is_singular(...) ) {ทำ} → if ( ! is_singular(...) ) {return;} · if ( ! is_admin() ) {ทำ} → if ( is_admin() ) {return;}
เปรียบเปรย: แบบซ้อน = พาคนที่ผ่านเกณฑ์เดินเข้าห้องซ้อนห้อง · แบบ guard = ยามยืนหน้าประตู ใครไม่เข้าเกณฑ์เชิญกลับทันที คนที่ผ่านเดินเข้าโถงใหญ่ตรงๆ
⚠️ ทำไมต้องมีปีกกา { } แม้มีบรรทัดเดียว: WordPress Coding Standards (WPCS) ห้าม เขียน if แบบไม่มีปีกกา (กฎ Generic.ControlStructures.InlineControlStructure) — ถ้ารัน phpcs --standard=WordPress (จะตั้งใน Week 9) จะโดนฟ้อง · เหตุผลเชิงวิศวกรรม: วันหน้าเติมบรรทัดที่ 2 เข้าไปแล้วลืมใส่ปีกกา = บั๊กเงียบที่หายากมาก
ข้อยกเว้นที่จะเจอในโค้ดจริง: บรรทัดกัน direct access if ( ! defined( 'ABSPATH' ) ) exit; นิยมเขียนบรรทัดเดียวกันทั้งวงการ (เจอในไฟล์จริงของโปรเจคด้วย) — ถ้าอยากให้ผ่าน WPCS ใช้รูปแบบ defined( 'ABSPATH' ) || exit; ซึ่งไม่ใช่ control structure จึงไม่โดนกฎนี้

3. Callback เป็นฟังก์ชันมีชื่อ — ไม่ยัด logic ยาวใน closure

❌ closure ยาว — ถอด hook ทีหลังไม่ได้

add_filter('the_title', function($t){
  // logic ยาว 20 บรรทัด...
  // ใครอยาก remove_filter ทำไม่ได้
  // เพราะ closure ไม่มีชื่อให้อ้าง
});

✅ ฟังก์ชันมีชื่อ — ถอดได้ debug ง่าย

add_filter('the_title', 'whl_fire_title');
function whl_fire_title( $title ) {
  // Query Monitor โชว์ชื่อนี้ตรงๆ
  return $title;
}
กฎทอง: เขียนให้ "คนถัดไปอ่านรู้เรื่องใน 10 วินาที" — closure สั้น 1 บรรทัดใช้ได้ แต่ logic จริงจังต้องมีชื่อเรียก เพราะชื่อคือ documentation ที่ไม่มีวันหลุด sync

สร้าง plugin — โฟลเดอร์ wp-content/plugins/wp-hooks-lab/ ไฟล์หลัก wp-hooks-lab.php พร้อม plugin header ถูกต้อง + กัน direct access (if ( ! defined( 'ABSPATH' ) ) exit; — ค้นดูว่าทำไมทุกไฟล์ WP มีบรรทัดนี้)
3
เสียบ hook 3 จุด (โจทย์บังคับ):
① action บน wp_head — เพิ่ม <meta name="developer" content="..."> เฉพาะหน้า front page (ต้องหา conditional tag เอง)
② filter บน the_title — เติม 🔥 หน้า title เฉพาะ post ที่เผยแพร่เกิน 30 วัน (ฝึกครบวงจร รับค่า-แปลง-คืนค่า + ใช้ข้อมูลของ post)
③ filter บน excerpt_length — เปลี่ยนความยาว excerpt เป็น 30 คำ
4
พิสูจน์ความ dynamic — สร้าง post ทดลอง 2-3 อัน แก้ title ใน wp-admin แล้ว refresh ดูหน้าเว็บเปลี่ยนเอง + ดูผล hook ทั้ง 3 ใน view-source / หน้า blog
5
README มาตรฐาน portfolio + push GitHub — ตาราง hook ที่ใช้ (จุดไหน / action หรือ filter / ทำไมเลือกจุดนั้น), วิธีติดตั้ง, AI Usage section (เขียนมือ 100%) · commit แยกขั้นแบบ feat: add front-page meta tag action

✅ Definition of Done

  • Plugin activate ได้ไม่มี error · hook ทำงานครบ 3 ข้อ · แก้ข้อมูลใน admin แล้วหน้าเว็บเปลี่ยนเอง (ข้อ 4)
  • ลองลืม return ใน filter ② หนึ่งครั้ง → ดูผลที่เกิด → เขียนสิ่งที่เห็นใส่ README (บั๊กนี้ต้องเจอกับตาตัวเอง)
  • เปิด plugin แล้วดูเมนูเว็บ + หน้ารายการ post ใน admin — 🔥 โผล่ที่ที่ไม่ตั้งใจไหม? เพราะอะไร? → เขียนสิ่งที่เจอ + วิธีกันใส่ README (คำใบ้: the_title ยิงทั้งใน admin และกับ nav menu — ดู is_admin(), in_the_loop())
  • README ครบ + commit history อ่านรู้เรื่อง + ตอบ Quiz ด้านล่างได้โดยไม่เปิดเนื้อหา

📝 Quiz — ทดสอบความเข้าใจ (ตอบเองก่อนค่อยกางเฉลย · ส่งคำตอบมาให้ผมตรวจได้)

ข้อ 1: ทีมคอนเทนต์แก้ชื่อร้านอาหารใน wp-admin — ข้อมูลถูกเขียนลงที่ไหน และทำไมหน้าเว็บถึงเปลี่ยนตามโดย dev ไม่ต้องแตะโค้ด?

เขียนลง wp_posts.post_title ในฐานข้อมูล — หน้าเว็บเปลี่ยนตามเพราะ template ไม่ได้เก็บข้อความไว้ในตัว มันมีแค่ "ช่องว่าง" (the_title()) ที่ดึงค่าจาก DB สดๆ ทุกครั้งที่มีคนเปิดหน้า — นี่คือแก่นของ dynamic: ข้อมูลอยู่ที่เดียว (DB) ทุกหน้าที่อ้างถึงเปลี่ยนพร้อมกัน

ข้อ 2: ใน SQL dump ของ fujisawa มี revision 505 แถวใน wp_posts — revision คืออะไร ทำไม WP เก็บเป็น "post"?

ประวัติการแก้ไขแต่ละครั้ง — WP เก็บเป็นแถวใน wp_posts (post_type=revision, post_parent ชี้ตัวจริง) ตามหลัก "ตู้เดียว": ไม่ต้องสร้างตารางใหม่ ใช้กลไก query/restore เดิมได้หมด

ข้อ 3: add_action กับ add_filter ต่างกันยังไง — อันไหน "ลืม return แล้วข้อมูลหายทั้งเว็บ" เพราะอะไร?

action = ทำงาน ณ จุดนั้น ไม่ต้องคืนค่า · filter = รับค่า→แปลง→ต้องคืนค่า เพราะ WP เอาค่าที่ return ไปใช้แสดงจริง — ลืม return = คืน null = ข้อมูลส่วนนั้นกลายเป็นค่าว่างทุกหน้าที่ผ่าน filter นี้

ข้อ 4: URL /sightseeing/oze-park/ — WP ไล่หาไฟล์อะไรบ้าง เรียงลำดับ? ถ้าไม่มีสักไฟล์จบที่ไหน?

single-sightseeing-oze-park.php → single-sightseeing.php → single.php → singular.php → index.php (ปลายทางสุดท้ายเสมอ — ทุก theme จึงต้องมี index.php)

ข้อ 5: หน้าแรกของ discover-fujisawa ใช้ templates/home.php ทั้งที่กฎบอกว่า front-page.php ชนะทุกอย่าง — เป็นไปได้ยังไง?

front-page.php ชนะก็ต่อเมื่อมีไฟล์นั้นจริง — theme fujisawa ไม่มี WP จึงไล่บันไดต่อ และหน้าแรกเป็น static Page ที่เลือก Page Template "Home" ใน Page Attributes ซึ่งทางลัดที่คนเลือกเองชนะ page-{slug}.php / page.php ในบันไดของ page — บันไดทำงานเสมอ แต่ทางลัดแทรกได้ และไฟล์ที่ไม่มีจริงถูกข้ามเฉยๆ

ข้อ 6: ข้อมูล "เวลาเปิด-ปิด" ที่กรอกผ่าน ACF ไปอยู่ตารางไหน และถ้าหน้าเว็บแสดงเวลาผิด จะไปไล่ดูที่ไหนก่อน?

wp_postmeta (key–value ผูก post_id) — ACF คือ UI สวยๆ ครอบ postmeta ไม่ใช่ที่เก็บข้อมูลชนิดใหม่ เวลาผิดให้ไล่: ค่าใน postmeta ถูกไหม (ดูใน admin/phpMyAdmin) → template ดึง key ถูกตัวไหม → มี filter อะไรแปลงค่ากลางทางไหม (Query Monitor ช่วยได้)

ข้อ 7 (security/production): ทีม fujisawa สร้างตาราง wp_fjs_contact เก็บข้อมูลฟอร์มติดต่อเอง แทนที่จะเก็บใน wp_posts — เกณฑ์ตัดสินคืออะไร และถ้าเก็บใน wp_posts จะเสี่ยงอะไร?

เกณฑ์: คนทำคอนเทนต์ต้องแก้ผ่าน admin → wp_posts / ข้อมูลระบบ → ตารางเอง · ความเสี่ยงถ้าเก็บฟอร์มติดต่อใน wp_posts: ข้อมูลส่วนตัวผู้ใช้ (ชื่อ อีเมล ข้อความ) จะไหลเข้า UI ปกติของ WP — โผล่ใน search, REST API (ถ้า show_in_rest), sitemap ได้โดยไม่ตั้งใจ = ข้อมูลรั่ว · แยกตารางแล้วคุมการเข้าถึงเองชัดกว่า
