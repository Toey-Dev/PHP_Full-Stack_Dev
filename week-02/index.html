หลักการ 1: ปลั๊กอินไม่ใช่เวทมนตร์ — มันคือ "UI ครอบโค้ด"

ปัญหาที่มันเกิดมาแก้: การสร้างชนิดข้อมูลใหม่ใน WordPress ต้องเขียนโค้ด PHP (register_post_type()) แต่คนที่ดูแลเว็บส่วนใหญ่ — เจ้าของธุรกิจ, ทีมคอนเทนต์, นักการตลาด — เขียน PHP ไม่ได้ ปลั๊กอินอย่าง CPT UI เกิดมาเพื่อเป็นฟอร์มหลังบ้านที่แปลง "การคลิกในหน้าจอ" เป็น "โค้ดที่ WordPress ต้องการ" ให้อัตโนมัติ

เปรียบเปรย: ปลั๊กอิน = เครื่องทำกาแฟแคปซูล · เบื้องหลังยังเป็นการต้มกาแฟแบบเดิม (บดเมล็ด อัดแรงดัน) แต่แทนที่จะให้คุณทำเองทุกขั้น มันซ่อนความยุ่งยากไว้หลังปุ่มเดียว — CPT UI ก็ซ่อน register_post_type() ไว้หลังฟอร์ม สิ่งสำคัญคือ: เมื่อรู้ว่าข้างในคือการต้มกาแฟธรรมดา วันหนึ่งคุณก็ต้มเองด้วยมือได้ (เขียนโค้ดเอง) — นั่นคือเป้าของบทนี้

สูตรที่ใช้ได้กับปลั๊กอินทุกตัว: ปลั๊กอิน = โค้ด PHP (ที่เสียบ hook เหมือนเรา) + หน้าจอตั้งค่า + ที่เก็บ config ใน DB

หลักการ 2: ทุกปลั๊กอินเสียบ hook ที่เรารู้จักแล้วจาก Week 1

จำ "รูปลั๊กไฟตามไทม์ไลน์" จาก Week 1 ได้ไหม? ปลั๊กอินทุกตัวก็เสียบปลั๊กที่รูเดียวกันนั้น:

ปลั๊กอินเสียบที่ hookทำอะไร
CPT UIinitอ่าน config จาก DB แล้วเรียก register_post_type() / register_taxonomy()
ACFacf/init, edit_form_*วาด metabox ในหน้าแก้โพสต์ + เซฟค่าลง wp_postmeta
qTranslate-XTfilter หลายจุด (the_title, the_content...)ตัดภาษาที่ถูกต้องออกจากช่อง [:en]…[:th]… ตาม URL
AIOSEOwp_headพ่น meta description + Organization schema เข้า <head>

ไม่มีอะไรที่ปลั๊กอินทำได้แต่เราทำไม่ได้ — ต่างแค่มันมีหน้าจอสวยให้คนอื่นใช้

หลักการ 3: เมื่อไหร่ใช้ปลั๊กอิน / เมื่อไหร่เขียนเอง / ใส่ไว้ที่ไหน

เกณฑ์ทองที่จะตอบทั้งบท (มาจากเอกสารส่งมอบจริง): logic ผูกกับหน้าตา → ใส่ theme · logic ที่ข้อมูลต้องอยู่รอดแม้เปลี่ยน theme → ใส่ plugin

นี่คือเหตุผลที่ CPT ต้องอยู่ใน plugin ไม่ใช่ theme — ถ้า register CPT ไว้ใน functions.php ของ theme วันที่เปลี่ยน theme เนื้อหา 2,000+ ชิ้นของ Fujisawa จะ "หายไปจากเมนู admin" ทันที (ข้อมูลยังอยู่ใน DB แต่ WP ไม่รู้จัก post_type นั้นแล้ว = เข้าไม่ถึง) Counter-example นี้คือฝันร้ายที่ทำให้ทีมมืออาชีพแยก CPT ออกเป็น plugin เสมอ

ส่วนที่ 1: CPT UI ทำงานยังไง (ตัวสร้าง "ชนิดข้อมูล")

เมื่อวีกดสร้าง post type ใน CPT UI แล้วเห็นเมนูใหม่โผล่ — เบื้องหลังคือโค้ดนี้ทำงานตอน hook init:

add_action('init', function () {                    // 1
  register_post_type('whattosee', [                 // 2
    'labels'      => ['name' => 'สถานที่ท่องเที่ยว'], // 3
    'public'      => true,                           // 4
    'has_archive' => true,                           // 5
    'rewrite'     => ['slug' => 'sightseeing'],      // 6
    'supports'    => ['title','editor','thumbnail'], // 7
    'show_in_rest'=> true,                            // 8
  ]);
});
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
1เสียบที่ init — เพราะ register ต้องเกิดทุกครั้งที่โหลดหน้า (CPT ไม่ได้ถูกจำถาวรใน DB · WP "ลืม" มันทุก request แล้วประกาศใหม่) init คือจุดที่ WP พร้อมรับการ register พอดี
2ชื่อ slug whattosee = ค่าในคอลัมน์ post_type ของ wp_posts (เชื่อมกลับ Week 1!) — นี่คือ "ป้ายลิ้นชัก" ที่แยกเนื้อหาชนิดนี้ออกจากชนิดอื่น
3labels = ข้อความที่โผล่ในเมนู admin — เปลี่ยนแค่หน้าตา ไม่กระทบข้อมูล
4public => true = ให้มีหน้าเว็บสาธารณะ + โผล่ในผลค้นหา · ถ้า false = ข้อมูลระบบที่ไม่ให้คนนอกเห็น
5has_archive => true = เปิดหน้ารวม /sightseeing/ อัตโนมัติ — ของจริง Fujisawa มีแค่ 4/18 CPT ที่ตั้ง true อีก 14 ตัวต้องทำ listing เอง (Week 3 + 6 เจาะ)
6rewrite slug = เปลี่ยน URL จาก /whattosee/ เป็น /sightseeing/ — ต้อง flush permalinkหลังแก้ ไม่งั้น 404 (กับดักคลาสสิก)
7supports = เปิด/ปิดช่องในหน้าแก้โพสต์ · ทุก CPT ของ Fujisawa เปิดแค่ title/editor/thumbnail → เนื้อหาเชิงลึกทั้งหมดจึงอยู่ใน ACF (โยง Week 4)
8show_in_rest => true = ดึงผ่าน REST API + ใช้ Gutenberg ได้ · ⚠️ มุม security: เปิด = เพิ่ม attack surface (ใครก็ query โครงสร้างข้อมูลผ่าน /wp-json/ ได้) — ต้องรู้ตัวว่าเปิดอะไรออกไป
ลองพิสูจน์เอง: CPT UI มีปุ่ม Tools → Get Code ที่ export โค้ด register_post_type() ตรงๆ จาก config ที่ตั้งไว้ — เปิดดูแล้วจะเห็นว่า "หน้าจอที่คลิก" กับ "โค้ดที่เราเขียน" คือสิ่งเดียวกัน

ส่วนที่ 2: ACF ทำงานยังไง (ตัวสร้าง "ช่องกรอกข้อมูล")

เปรียบเปรย: ถ้า CPT คือ "ตู้เอกสารเปล่า" ACF คือช่างที่มาติดช่องกรอกเพิ่มที่หน้าแฟ้ม — ช่อง "เวลาเปิด-ปิด", "พิกัด", "FAQ" ที่ทีมคอนเทนต์กรอก แล้วมันแอบเก็บลง wp_postmeta (จำโพสต์อิทจาก Week 1 ได้ไหม)

เส้นทางข้อมูลของ ACF มี 4 สถานี:

สถานีเกิดอะไร
1. Field Group + Location Ruleบอกว่า "ช่องกรอกชุดนี้โผล่ที่ CPT ไหน" (เช่น location = post_type whattosee)
2. คนกรอกใน adminACF วาด metabox ให้กรอก แล้วเซฟลง wp_postmeta เป็นคู่ key–value
3. get_field('opening_hours')template ดึงค่ากลับมา — ข้างใต้คือ get_post_meta() แต่ ACF แปลง type ให้พร้อมใช้
4. escape + แสดงecho esc_html(...) — Week 4 เจาะเต็ม
🔍 Why get_field() ไม่ใช่ get_post_meta(): field ชนิด image ถ้าใช้ get_post_meta() ได้แค่ตัวเลข ID ดิบ · แต่ get_field() คืน array พร้อม URL/alt/ขนาดต่างๆ ให้เลย — ACF คือ "postmeta ที่แต่งตัวมาแล้ว" · แต่ต้องรู้ว่าข้างใต้คือ postmeta เสมอ เผื่อวันที่ต้อง debug หรือถอด ACF ออก

ส่วนที่ 3: qTranslate + AIOSEO (2 ปลั๊กอินที่เสียบ filter/action)

qTranslate-XT = ล่ามที่ยืนหน้าประตู: ทีมคอนเทนต์กรอกทุกภาษาในช่องเดียว [:en]Sea Candle[:th]ประภาคาร[:] แล้ว qTranslate เสียบ filter ที่ the_title/the_content คอย "ตัดเฉพาะภาษาที่ URL ขอ" ออกมาก่อนแสดง — คนดูหน้า /th/ เห็นแค่ภาษาไทย ไม่เห็นแท็ก (Week 5 เจาะเต็ม + มีบั๊กจริงรอวีแก้)

AIOSEO = เลขาที่แปะป้ายให้เครื่องอ่าน: เสียบ action ที่ wp_head พ่น <meta name="description"> + Organization schema เข้าหัวเว็บทุกหน้า — เหมือน inc/schema.php ของ wlf ที่เราอ่านใน Week 1 แค่ทำเป็นปลั๊กอินสำเร็จรูป

ประเด็นสำคัญของงาน redesign: AIOSEO ทำ schema พื้นฐาน (Organization/meta) ให้ แต่ schema เชิงลึกอย่าง TouristAttraction + FAQPage + GeoCoordinates ที่ผูกกับ ACF ต้องเขียนเองใน template (Week 8) — รู้ว่าปลั๊กอินทำถึงไหน เราต่อจากตรงไหน คือทักษะสำคัญ

ส่วนที่ 4: อ่านโค้ดจริง — functions-discover-fujisawa.php ทีละฟังก์ชัน

นี่คือไฟล์ dev-starter จริงของโปรเจค (อยู่ใน dev-starter/) — โค้ดที่วีจะใช้จริง ไฟล์ยาว ~250 บรรทัด แบ่งเป็น 3 ก้อนงาน ไล่อ่านทีละฟังก์ชัน (โค้ดใหญ่ไม่อ่านทีละบรรทัด — ดูภาพรวมของแต่ละก้อนว่าทำอะไร ทำไม):

📌 อ่านโค้ดจริงต้องแยกให้ออก: "สไตล์ที่เขาเขียน" ≠ "สไตล์ที่เราควรเขียน" — โค้ดด้านล่างยกมาจากไฟล์จริงแบบไม่ดัดแปลง จะเห็น if (...) exit; / if (...) return; เขียนบรรทัดเดียวไม่มีปีกกา ซึ่ง WPCS ฟ้อง (ดูกฎใน Week 1 · Clean Code ข้อ 2) · อ่านของเขาให้เข้าใจ แต่เวลาเขียนเองใส่ปีกกาเสมอ — นี่คือทักษะที่จะใช้เต็มๆ ใน Week 7 ตอน refactor theme เก่า

ก้อนที่ 1 — add_action('init', ...): register CPT + taxonomy

if (!defined('ABSPATH')) exit;   // กันเปิดไฟล์ตรงๆ จากเบราว์เซอร์

add_action('init', function () {
  register_post_type('whattosee', [ /* ...args... */ ]);   // สถานที่
  register_post_type('news',      [ /* ...args... */ ]);   // ข่าว

  register_taxonomy('spot_cat', 'whattosee', [ /* ... */ ]);      // ประเภทสถานที่
  register_taxonomy('news_category', 'news',  [ /* ... */ ]);     // หมวดข่าว
});

ฟังก์ชันนี้ทำอะไร: ประกาศชนิดข้อมูล 2 ตัว (whattosee, news) + ป้ายจัดหมวด 2 ตัว ตอน init · ทำไมรวมทุกอย่างใน action เดียว: เพราะทั้ง CPT และ taxonomy ต้อง register ที่ init เหมือนกัน — จับกลุ่มไว้ด้วยกันอ่านง่าย · จุดสังเกต: register_taxonomy('spot_cat', 'whattosee', ...) ผูก taxonomy เข้ากับ CPT ตรงพารามิเตอร์ที่ 2 (Week 3 เจาะว่าทำไม taxonomy ถึงแชร์ข้าม CPT ได้)

NOTE จากไฟล์จริง (สำคัญมาก): คอมเมนต์ในไฟล์เตือนว่า "production มี CPT อยู่แล้ว — ของจริงให้ตั้งผ่าน CPT UI แล้วเพิ่มเฉพาะฟิลด์ GEO ที่ขาด ไม่ต้อง re-register ทับ" → โค้ด register นี้ไว้เรียนรู้ + สำหรับ theme เปล่า · ถ้าเผลอ register ทับ CPT ที่ CPT UI ทำไว้ = ตั้งค่าชนกัน เว็บพัง

ก้อนที่ 2 — add_action('acf/init', ...): ประกาศ ACF field ด้วยโค้ด

add_action('acf/init', function () {
  if (!function_exists('acf_add_local_field_group')) return;  // กัน error ถ้าไม่มี ACF

  acf_add_local_field_group([
    'key'      => 'group_df_spot_geo',
    'title'    => 'สถานที่ — ข้อมูล GEO/AEO',
    'location' => [[['param'=>'post_type','operator'=>'==','value'=>'whattosee']]],
    'fields'   => [ /* summary_th, opening_hours, geo_lat, faq(repeater)... */ ],
  ]);
});

ฟังก์ชันนี้ทำอะไร: สร้าง Field Group ด้วยโค้ดแทนการคลิกใน ACF UI · ทำไมต้องมีบรรทัด if (!function_exists(...)): guard clause (จำจาก Week 1!) — ถ้าปลั๊กอิน ACF ถูกปิด ฟังก์ชันนี้จะไม่มีอยู่ เรียกไปเว็บล่ม เช็คก่อนเรียกจึงปลอดภัย · location คืออะไร: "Location Rule" = กติกาว่า field ชุดนี้โผล่ที่ไหน — ตรงนี้บอกว่า "โผล่เมื่อ post_type == whattosee" · ทำไมประกาศ ACF ด้วยโค้ดแทน UI: เพื่อ version control (เข้า git เห็น diff ได้) และ deploy ไปกับ theme — Week 4 เจาะเรื่อง acf-json

ก้อนที่ 3 — df_get_banner_emergency(): helper ดึงข่าวฉุกเฉิน

function df_get_banner_emergency() {
  $now = current_time('Y-m-d H:i:s');
  $q = new WP_Query([
    'post_type'  => 'news',
    'meta_query' => [ /* display_in_banner=1 AND category=emergency AND ยังไม่หมดอายุ */ ],
  ]);
  if ($q->have_posts()) { /* ...คืน text + url... */ }
  return null;
}

ฟังก์ชันนี้ทำอะไร: ค้นข่าวที่ตั้งธง "ฉุกเฉิน + ให้ขึ้นแบนเนอร์ + ยังไม่หมดอายุ" มา 1 ข่าว เพื่อเอาไปโชว์แถบแดงบนหัวเว็บ · ทำไมเป็น helper แยก: เพราะ header.php เรียกใช้ได้จากที่เดียว คุมง่าย (Week 6 จะเอาไปใช้จริง) · ตอนนี้ยังไม่ต้องเข้าใจ meta_query ลึก — Week 6 (WP_Query) จะผ่าทีละบรรทัด รวมถึงกับดักจริงในนั้น · ตอนนี้ขอแค่เห็นว่า "helper = ฟังก์ชันของเราที่ห่อ query ไว้ให้เรียกซ้ำได้"

ตัวอย่างประยุกต์

🏢 งานวี: ใน stack ของ Fujisawa มีปลั๊กอินชื่อ "Fujisawa migration system" ที่ทีมเขียนเอง เพื่อสร้างตาราง wp_import_* 18 ตัวและย้ายข้อมูล — นี่คือหลักฐานตรงหน้าว่า "เขียน plugin เองได้จริง และทีมวีก็ทำอยู่แล้ว" ปลั๊กอินไม่ใช่ของที่ต้องซื้อ/ดาวน์โหลดเท่านั้น เมื่อเข้าใจว่ามันคือโค้ด PHP + hook เราสร้างเองได้ตามต้องการ

🌏 อุตสาหกรรม: WooCommerce (ปลั๊กอิน e-commerce อันดับ 1) ก็แค่ปลั๊กอินตัวใหญ่ที่ register_post_type('product') + register taxonomy product_cat + เพิ่ม meta ราคา — โครงเดียวกับที่วีกำลังจะเขียนใน Mission เป๊ะ ต่างแค่ scale

🧹 Clean Code ประจำ Week 2 — เขียน register แบบมืออาชีพ

1. แยกไฟล์ register ออกจาก functions.php

📝 Quiz — ทดสอบความเข้าใจ

ข้อ 1: ทำไม register_post_type() ต้องเรียกที่ hook init ทุก request ไม่เก็บถาวรใน DB ทีเดียว?

เพราะ WordPress "ลืม" CPT ทุกครั้งที่จบ request — โค้ด PHP รันใหม่หมดทุกครั้งที่มีคนเปิดหน้า CPT ไม่ใช่ข้อมูลใน DB แต่เป็น "การประกาศ" ที่ต้องทำใหม่ทุกรอบ · init คือจังหวะที่ WP โหลดเสร็จพอจะรับ register แล้ว (สิ่งที่เก็บถาวรใน DB คือตัวโพสต์ใน wp_posts — ไม่ใช่นิยามของ post type)

ข้อ 2: CPT UI กับการเขียน register_post_type() เอง ให้ผลต่างกันไหม? แล้วปลั๊กอินมีข้อดีอะไร

ผลลัพธ์เหมือนกันเป๊ะ (CPT UI พิสูจน์เองด้วยปุ่ม Get Code) — ต่างที่ CPT UI ให้คนไม่เขียนโค้ดตั้งค่าได้ผ่านฟอร์ม + เก็บ config ใน DB ส่วนเขียนโค้ดเองได้ version control + deploy ไปกับ theme/plugin + ปรับแต่งลึกกว่า · เลือกตามทีม: มีคนแก้ที่ไม่ใช่ dev → UI / อยาก track ใน git → โค้ด

ข้อ 3 (security/production): CPT ทุกตัวของ Fujisawa ตั้ง show_in_rest => true — สิ่งนี้กระทบ attack surface ยังไง?

เปิด REST = ใครก็ได้ยิง /wp-json/wp/v2/{cpt} เพื่อดูโครงสร้าง/รายการข้อมูลได้โดยไม่ต้อง login (สำหรับ public CPT) — เพิ่มพื้นที่ให้ถูกสำรวจ (enumeration) · ไม่ใช่ช่องโหว่โดยตรง แต่ต้องรู้ว่าเปิดอะไรออกไป: ข้อมูลที่ไม่อยากให้ดึงง่ายๆ ควรพิจารณาปิด REST หรือจำกัดสิทธิ์ (โยง Week 4: ACF field ก็มี show_in_rest แยกอีกชั้น) · หลักคิด: เปิดเท่าที่จำเป็น (least exposure)

ข้อ 4: NOTE ในไฟล์ dev-starter เตือนว่า "อย่า re-register ทับ CPT ที่ production มีอยู่แล้ว" — ทำไมถึงอันตราย?

ถ้า CPT UI ตั้ง whattosee ไว้แล้ว แล้วเราเขียนโค้ด register whattosee ซ้ำด้วย args คนละชุด = ตั้งค่าชนกัน (เช่น slug/has_archive/supports ไม่ตรง) → พฤติกรรมกลายเป็นอันไหนโหลดทีหลังชนะ คาดเดายาก · หน้าอาจ 404 หรือ field หาย · ทางถูก: production ใช้ CPT UI อยู่แล้ว → เราแค่ "เพิ่มเฉพาะ ACF field ที่ขาด" ไม่แตะการ register CPT เดิม
