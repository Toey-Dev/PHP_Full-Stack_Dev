🧠 แนวคิด & หลักการ

หลักการ 1: GEO/AEO — เขียนเว็บให้ "เครื่องอ่าน" ไม่ใช่แค่ "คนอ่าน"

ปัญหาที่มันแก้: ยุคนี้คนหาข้อมูลผ่าน AI ("ที่เที่ยว Fujisawa มีอะไรบ้าง?") — AI ไม่ได้เปิดเว็บมาอ่านสวยๆ แบบมนุษย์ มันต้องการข้อมูลที่มีโครงสร้างชัดให้ดึงไปตอบได้แม่นยำ · GEO (Generative Engine Optimization) + AEO (Answer Engine Optimization) คือการทำเว็บให้ AI "เข้าใจและยกไปตอบ" — ต่อยอดจาก SEO เดิม

เปรียบเปรย: JSON-LD = ป้ายราคา/ฉลากสินค้าที่เขียนเป็นบาร์โค้ด · คนดูรูปกับชื่อก็รู้ว่าเป็นอะไร แต่เครื่องสแกน (AI/crawler) อ่านบาร์โค้ดได้ข้อมูลครบถ้วนแม่นยำกว่า — ราคา น้ำหนัก แหล่งผลิต · JSON-LD คือ "บาร์โค้ด" ที่ซ่อนในหน้าเว็บให้เครื่องอ่าน

หลักการ 2: 🔴 crawler ไม่รัน JavaScript — schema ต้องมากับ HTML ตั้งแต่ต้น

นี่คือหลักที่พลาดไม่ได้ และเป็นบทเรียนจากความผิดพลาดจริงของโปรเจค:

💥 เหตุการณ์จริง (docs/03): เครื่องมือตรวจ GEO ชื่อ Otterly เคยพบว่าหน้า /th/sightseeing/ ของ Fujisawa render ด้วย JavaScript → Static score = 0% = AI/crawler อ่านไม่ได้เลย (เห็นหน้าเปล่า) · แม้เนื้อหาจะโผล่ครบเมื่อคนเปิดดู เพราะcrawler ส่วนใหญ่ไม่รัน JS — มันขอ HTML แล้วอ่านทันที ถ้าเนื้อหา/schema มาทีหลังด้วย JS = มองไม่เห็น

เปรียบเปรย: ใส่ schema ด้วย JS = ส่งจดหมายที่หมึกจะปรากฏหลังเปิดซอง 5 นาที · คนใจเย็นรอได้ แต่ไปรษณีย์ (crawler) ที่สแกนซองแล้วส่งต่อทันทีเห็นซองเปล่า · กฎเหล็ก: JSON-LD ต้อง render ด้วย PHP ฝั่ง server — อยู่ใน HTML ตั้งแต่ครั้งแรกที่ตอบกลับ

หลักการ 3: summary_th ต้องอยู่บนสุด — "Quick-Answer" ที่ AI ยกไปตอบ

โยง Week 4: field summary_th (2-3 ประโยคตอบ "ที่นี่คืออะไร น่าไปไหม") ไม่ใช่แค่กล่องสรุปสวยๆ — มันคือคำตอบสำเร็จรูปที่ AI จะดึงไปตอบผู้ใช้โดยตรง · จึงต้องอยู่บนสุดของหน้า (ทั้งใน HTML และใน schema description) · หลักคิด AEO: เขียนคำตอบให้ AI ยกไปใช้ได้เลย ไม่ต้องให้มันเดา

ส่วนที่ 2: อ่านโค้ดจริง — สร้าง JSON-LD ใน single-whattosee.php

dev-starter เลือกวิธี "สร้าง PHP array แล้ว wp_json_encode()" — ปลอดภัยกว่าการ echo JSON ด้วยมือ · อ่านทีละส่วน:

$graph = [                                       // 1
  [
    '@type'       => 'TouristAttraction',        // 2
    'name'        => get_the_title(),            // 3
    'description' => $summary,                    // 4 ← summary_th (Quick-Answer)
    'image'       => get_the_post_thumbnail_url(null,'large'),
    'address'     => [
      '@type' => 'PostalAddress',
      'addressLocality' => 'Fujisawa',
      'addressRegion'   => 'Kanagawa',
      'addressCountry'  => 'JP',
    ],
  ],
];
if ($lat && $lng) {                               // 5
  $graph[0]['geo'] = [
    '@type' => 'GeoCoordinates',
    'latitude' => $lat, 'longitude' => $lng,
  ];
}
if ($faq) {                                      // 6
  $items = [];
  foreach ($faq as $row) {
    $items[] = [
      '@type' => 'Question',
      'name'  => $row['q'],
      'acceptedAnswer' => ['@type'=>'Answer','text'=>$row['a']],
    ];
  }
  $graph[] = ['@type'=>'FAQPage','mainEntity'=>$items];  // 7
}

$ld = ['@context' => 'https://schema.org', '@graph' => $graph];  // 8
echo '<script type="application/ld+json">'
   . wp_json_encode($ld, JSON_UNESCAPED_UNICODE | JSON_UNESCAPED_SLASHES)  // 9
   . '</script>';
บรรทัดทำไมต้องมี (ไม่ใช่แค่ทำอะไร)
1-2เริ่ม graph ด้วย TouristAttraction — เป็น PHP array ธรรมดา (ยังไม่ใช่ JSON) เพื่อให้ต่อเติม/เช็คเงื่อนไขได้ง่าย
3-4ทุกค่าดึงจาก DB (get_the_title() ผ่าน qTranslate · $summary = ACF) — ไม่ hardcode · description = summary_th ที่ AI ยกไปตอบ
5-6เติม geo/faq เฉพาะเมื่อมีข้อมูล (guard clause — Week 4) · ถ้าไม่มีพิกัดแล้วใส่ geo ว่าง = schema error
7เพิ่ม FAQPage เป็นสมาชิกใหม่ใน graph — หลาย schema type อยู่ในหน้าเดียวได้ผ่าน @graph
8ห่อด้วย @context + @graph = "schema เดียวต่อหน้า"
9หัวใจ: wp_json_encode() สร้าง JSON ที่ถูก syntax เสมอ (ไม่มี trailing comma, escape ถูก) · JSON_UNESCAPED_UNICODE = ภาษาไทย/ญี่ปุ่นไม่กลายเป็น \u... อ่านออก

ส่วนที่ 3: 2 วิธีเขียน JSON-LD — ทำไม array ปลอดภัยกว่า echo มือ

❌ echo JSON ด้วยมือ (docs/03 แบบ copy-paste)

echo '{
  "@type": "FAQPage",
  "mainEntity": [';
foreach ($faq as $row) {
  echo '{ "name": "'
    . esc_js($row['q']) . '" },';  // ← comma!
}
echo ']}';

เสี่ยง: trailing comma หลัง element สุดท้าย = JSON พัง · ต้อง esc_js() ทุกค่าเอง · ลืมจุดเดียว schema ทั้งหน้าใช้ไม่ได้

✅ สร้าง array → wp_json_encode

$items = [];
foreach ($faq as $row) {
  $items[] = [
    'name' => $row['q'],  // ไม่ต้องห่วง comma
  ];
}
echo wp_json_encode(
  ['mainEntity' => $items]
);

PHP จัดการ comma/escape/syntax ให้หมด · เพิ่ม/ลบ field ไม่พัง · อ่านง่าย debug ง่าย

กฎทอง: อย่าประกอบ JSON เป็น string ด้วยมือถ้าเลี่ยงได้ — ให้ภาษาสร้างให้ (PHP=wp_json_encode, JS=JSON.stringify) · string manipulation กับ JSON = บั๊ก syntax ที่หายากและ schema พังเงียบ (Google อ่านไม่ออกแต่ไม่มี error ในหน้า)

ส่วนที่ 4: ⚠️ qTranslate กับ JSON-LD (โยง Week 5)

จำกับดักจาก Week 5 ได้ไหม — ข้อความหลายภาษาที่ไม่ผ่าน filter จะโผล่แท็กดิบ [:th]... · ใน JSON-LD ต้องระวังเป็นพิเศษ:

  • ผ่าน filter อัตโนมัติ: get_the_title(), get_field() — ตัดภาษาให้แล้ว ใส่ใน graph ได้เลย
  • ต้องจัดการเอง: ถ้าดึงข้อความหลายภาษาแบบ raw มาประกอบ ต้อง qtranxf_use_language() ก่อน ไม่งั้น AI อ่านเจอ [:en]Sea Candle[:th]... ทั้งก้อน = ข้อมูลขยะเข้า schema
ทำไมร้ายกว่าปกติ: แท็กดิบบนหน้าเว็บคนเห็นแล้วรู้ว่าบั๊ก · แต่ในJSON-LD ที่ซ่อนอยู่ คนไม่เห็น — AI ดูดข้อมูลเสียไปตอบผู้ใช้โดยเราไม่รู้ตัว · ต้องทดสอบด้วย Rich Results Test + ดู view-source ทุกภาษา

🧹 Clean Code ประจำ Week 8 — สร้าง schema ให้สะอาด

1. สร้าง array แล้ว encode — ไม่ต่อ string JSON เอง

❌ heredoc JSON — เปราะ escape ยาก

echo "<script>{
  \"name\": \"" . $name . "\",
  \"lat\": " . $lat . "
}</script>";
// $name มี " = พัง · ลืม escape

✅ array → wp_json_encode

$data = ['name' => $name, 'lat' => $lat];
echo '<script type="application/ld+json">'
  . wp_json_encode($data)
  . '</script>';
// escape/comma/quote ปลอดภัยหมด

2. เติม field เฉพาะที่มีค่า — ไม่ยัด null เข้า schema

❌ ใส่ทุก field แม้ว่าง = schema error

$g['geo'] = [
  'latitude'  => $lat,  // ว่าง = null
  'longitude' => $lng,
];
// ไม่มีพิกัด → geo ว่าง → validator ฟ้อง

✅ guard clause ก่อนเติม

if ($lat && $lng) {
  $g['geo'] = [
    'latitude'=>$lat,'longitude'=>$lng
  ];
}
// ไม่มีข้อมูล = ไม่ใส่ key เลย
กฎทอง: schema ที่ดี = "มีเฉพาะข้อมูลที่จริงและครบ" — ใส่ field ว่าง/เดาๆ ทำให้ validator ฟ้องหรือ AI ได้ข้อมูลผิด · น้อยแต่ถูก ดีกว่ามากแต่มั่ว

📝 Quiz — ทดสอบความเข้าใจ

ข้อ 1: ทำไม JSON-LD ต้อง render ฝั่ง server (PHP) ห้ามใส่ด้วย JavaScript?

เพราะ crawler/AI ส่วนใหญ่ไม่รัน JavaScript — มันขอ HTML แล้วอ่านทันที · ถ้า schema มากับ JS (เติมทีหลัง) crawler เห็นหน้าเปล่า = อ่านไม่ได้ · เหตุการณ์จริง: Otterly พบ /th/sightseeing/ render ด้วย JS → Static 0% = AI มองไม่เห็นเว็บเลย · schema ต้องอยู่ใน HTML ตั้งแต่ response แรก

ข้อ 2: ทำไมสร้าง PHP array แล้ว wp_json_encode ปลอดภัยกว่า echo JSON เป็น string เอง?

เพราะ wp_json_encode() จัดการ syntax ให้ถูกเสมอ: ไม่มี trailing comma, escape quote/อักขระพิเศษถูก, โครงสร้างถูก · การต่อ string เองเสี่ยง trailing comma หลัง element สุดท้าย (JSON พัง) + ลืม escape ค่าที่มี " · schema ที่ syntax พังจะอ่านไม่ออกโดยไม่มี error เตือนในหน้า — หายากมาก

ข้อ 3 (security/production): ข้อความหลายภาษาที่ไม่ผ่าน qTranslate filter หลุดเข้า JSON-LD — ทำไมร้ายกว่าหลุดบนหน้าเว็บ?

บนหน้าเว็บคนเห็นแท็กดิบ [:th]... แล้วรู้ทันทีว่าบั๊ก แจ้งแก้ได้ · แต่ใน JSON-LD ซ่อนอยู่ (คนไม่เห็น) → AI/crawler ดูดข้อมูลที่มีแท็กดิบไปตอบผู้ใช้โดยเราไม่รู้ตัว = ข้อมูลขยะกระจาย · ป้องกัน: ใช้ get_field()/the_title() (ผ่าน filter) หรือ qtranxf_use_language() ถ้าประกอบเอง + ทดสอบ Rich Results ทุกภาษา

ข้อ 4: field summary_th ควรอยู่ตรงไหนของหน้าและ schema? เพราะอะไร (หลัก AEO)?

อยู่บนสุดของหน้า (HTML) และเป็น description ใน schema · เพราะมันคือ "Quick-Answer" — คำตอบสำเร็จรูป 2-3 ประโยคที่ AI จะยกไปตอบผู้ใช้โดยตรง ("ที่นี่คืออะไร น่าไปไหม") · หลัก AEO: เขียนคำตอบให้ AI หยิบใช้ได้ทันทีไม่ต้องเดา + วางตำแหน่งเด่นให้ทั้งคนและเครื่องเจอก่อน
